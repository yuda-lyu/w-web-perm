# 使用者群組關聯流程

## 觸發

管理員（瀏覽器使用者，經 w-ui-loginout 登入取得有效 token、`connState='csLogin'`）於後台後可由兩個入口進入同一份 `users.cgrups`（JSON5 格式 `{群組名:{mode:'OR'|'AND',isActive:'y'|'n'}}`）的編輯，差別在進入視角與儲存路徑：

- **入口 A（使用者視角）**：於使用者清單頁（`LayoutContentUsers.vue`）某列 `cgrups` 欄的按鈕，點擊觸發 `$dg.showVeCgrupsById(id)`（`LayoutContentUsers.vue:227`），開啟群組關聯編輯對話框 `VeCgrups`。此對話框以「該使用者擁有哪些群組」為列，逐群組可勾選啟用、切 OR/AND 合併模式。對話框 Save 後僅 `resolve` 一個 `cgrups` JSON 字串回填至該使用者列（`LayoutContentUsers.vue:1137`），**不自行打 API**；實際寫入 DB 由使用者清單頁的 Save 統一送出。
- **入口 B（群組視角）**：於群組清單頁（`LayoutContentGrups.vue`）某列 `belongUsers` 欄的按鈕，點擊觸發 `$dg.showVeGrupBlngUsersById(id)`（`LayoutContentGrups.vue:220`），開啟群組成員編輯對話框 `VeGrupBlngUsers`。此對話框以「全部使用者」為列，逐使用者可勾選是否歸屬本群組、切 OR/AND。對話框 Save 後於對話框內**自行**重組各 user 的 `cgrups` 並 `$fapi.updateUsers` 直接寫入 DB（`VeGrupBlngUsers.vue:973`）。

兩入口寫入端點同為 `updateUsers`，經 hapi `verifyConn` → `checkToken(token)` 守門（`server/WWebPerm.mjs:1155,1171`），**token 無效 / 過期一律拒絕連線**；是否限定管理員由部署方的 `verifyBrowserUser` 決定（開發樣本限 `isAdmin='y'`，`srv.mjs`）。

    - 雙語：eng / cht 各一輪。

## 執行流程

兩入口共用同一份 `users.cgrups`，但儲存路徑分流：入口 A 對話框 Save 僅 `resolve` 一個 `cgrups` 字串回填前端列、**不打 API**（實際寫 DB 待使用者清單頁 Save）；入口 B 對話框 Save 於對話框內**自行** `$fapi.updateUsers` 直接寫 DB。以下分兩條 trace。

### 路徑 A：使用者視角（VeCgrups，resolve 回填字串）

## spec 規則摘要

- **權限**：兩入口寫入皆經 `updateUsers` → `verifyConn` → `checkToken` 守門（有效 token 才可寫）；是否限管理員由部署方 `verifyBrowserUser` 決定（開發樣本限 `isAdmin='y'`）。
- **同源異視角**：入口 A（使用者頁 `cgrups` 按鈕 → `VeCgrups`）與入口 B（群組頁 `belongUsers` 按鈕 → `VeGrupBlngUsers`）編輯同一份 `users.cgrups`（JSON5 `{群組名:{mode:'OR'|'AND',isActive:'y'|'n'}}`），差別在「以群組為列 vs 以使用者為列」。
- **儲存路徑差異（核心）**：
  - 入口 A 對話框 Save 僅 `vo.pm.resolve(cgrups)` 回傳字串、回填前端使用者列並設 `isModified=true`，**不自行打 API**；實際寫 DB 由使用者清單頁的 Save 統一以 diff 批次送出。
  - 入口 B 對話框 Save 於 `core()` 內重組各受影響 user 的 `cgrups`、`await saveUsers` → `$fapi.updateUsers` **直接寫 DB**，並於對話框內呈現 `userSaveUsersSuccess` / `userSaveUsersFail`。
- **粒度**：入口 A 無 Save 成敗 toast（成敗由使用者清單頁呈現），僅開啟 / 切換守門訊息；入口 B 有 `userSaveUsersEmpty` / `userSaveUsersSuccess` / `userSaveUsersFail` 操作結果 toast。
- **邊界**：
  - `cgrups` 只記錄 `isActive==='y'`（啟用）的群組；取消勾選（enable=n）即自重組後的 `cgrups` 移除該群組鍵（`VeGrupBlngUsers.vue:704`、`VeCgrups.vue` 彙整時略過 enable≠y）。
  - `mode` 僅接受 `OR` / `AND`，回填時非此二值一律補 `OR`。
  - 入口 B 以 `fltKpGrup` 過濾使用者 `cgrups` 內已不存在於現行 `grups` 的群組鍵，避免過時鍵殘留（`VeGrupBlngUsers.vue:555`）。
  - 後端對 `users` **不檢查 name 重複**（`WWebPerm.mjs:405` 註「users 可重複 name」）；`email` 不可空 / 不可重複，由 `ckKey('email')` 守門（`WWebPerm.mjs:411`）——本流程不改 email，故僅入口 B 寫入時隨列一併受此守門。
- **契約**：兩對話框 `clickSave` 皆採 `core() + finally`，`finally` 一處關 loading（`VeCgrups.vue:806`、`VeGrupBlngUsers.vue:1088`）；入口 B `updateUsers` 失敗由 `errTemp` 旗標短路後續、走 `userSaveUsersFail`。對話框 Close 一律 `pm.reject('close window')`，由各入口 `.catch` 接住。
