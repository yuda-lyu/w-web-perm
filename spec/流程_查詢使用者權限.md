# 查詢使用者權限流程

## 觸發

瀏覽器端應用（外部系統）欲查詢「自身」（持有 token 的當前使用者）權限時，呼叫前端 SDK：先以 `getPerm(url, tokenTar, opt)` 直接取得 `{ user, rules }`，或以 `perm()` 工廠的 `conn(url)` 載入後用 `active(key)` / `getUser()` / `getRules()` 查詢單一規則或快取結果。

兩者最終都對後端 `/api/getPerm?token={token}` 發出 GET 請求：`getPerm.mjs` 以 `axios.get` 將 `url` 內 `{token}` 佔位符替換為實際 `tokenTar` 後送出（`src/getPerm.mjs:25-32`）；`perm.mjs` 的 `conn` 接收已帶妥 token 的完整 `url`，以 `axios.get` 送出（`src/perm.mjs:43`）。

後端 `/api/getPerm` handler 自 `req.query.token` 取 token，依序經 `getAndVerifyClientUser`（反查 perm `users` 表 + `verifyClientUser`）→ `getGenUserAndRulesByUserId`（查 `grups` / `pemis` / `targets` 並算 `getUserRules`）後，以 `pm2resolve` 包成 `{ state, msg }` 回傳（`server/WWebPerm.mjs:813-885`）。**token 無效 / 非 client 使用者 / 查無使用者一律 reject，`state` 回 `'error'`**。

## 重要流程

> 本流程為 query-api 類型：請求成功僅回傳 JSON 結構（`{ user, rules }`），無瀏覽器 UI 終態，故各 E2E 以 API 回傳結構 / `state` / 權限值為斷言對象，**不含 pixel baseline**。

- **E2E-001**
  - title: 持有效 token 的使用者查詢自身權限取得 user 與 rules
  - description: 驗證瀏覽器應用以有效 client 使用者 token 呼叫 `getPerm(url, tokenTar)`，後端依 token 解出該使用者、合併其群組與權限規則後，回傳含 `user`（含 `grupsNames`）與 `rules` 陣列的物件。屬 golden 成功路徑，其餘失敗 / 轉換案例皆以此回傳結構為基準。
  - flow:
    - 測試資料：base seed 既有 client 使用者（有對應 `grups` / `pemis` / `targets` 關聯）及其有效 token。
    - 操作：以該使用者 token 呼叫 `getPerm(url, tokenTar)`，url 含 `token={token}` 佔位符。
    - 驗證：
      1. 語意：回傳值為物件，含 `user` 與 `rules` 兩鍵；`user.id` 對應該 token 使用者、`user` 含 `grupsNames` 字串；`rules` 為陣列、各元素含 `name` 與 `isActive`。
      2. 視覺：純 API 無 UI 終態，無 pixel baseline。
    - 雙語：本流程為 API、回傳不含 i18n 文案，僅跑一輪。
    - 清理：本案唯讀，不改 DB，無需清理。

- **E2E-002**
  - title: 以 perm() 工廠 conn 載入後用 active 查詢單一規則
  - description: 驗證瀏覽器應用以 `perm()` 取得 SDK，`conn(url)`（url 已帶妥 token）載入成功後，`getUser()` / `getRules()` 回傳與 E2E-001 相同的 `user` / `rules`，且 `active(key)` 對既有規則名回傳 `'y'`（規則 `isActive='y'`）或 `'n'`（其餘 / 不存在）。屬 SDK 快取查詢成功案例。
  - flow:
    - 測試資料：base seed client 使用者與有效 token；該使用者至少一條 `isActive='y'` 規則與一個不存在的規則名。
    - 操作：`let p = perm()` → `await p.conn(url)` → 呼叫 `p.getUser()` / `p.getRules()` / `p.active(知名規則名)` / `p.active(不存在規則名)`。
    - 驗證：
      1. 語意：`getUser()` 回傳物件含該使用者 `id`；`getRules()` 回傳陣列；`active(已啟用規則名)` 回 `'y'`；`active(不存在規則名)` 回 `'n'`。
      2. 視覺：純 API 無 UI 終態，無 pixel baseline。
    - 雙語：API 流程，僅跑一輪。
    - 清理：本案唯讀，不改 DB，無需清理。

- **E2E-003**
  - title: token 無效時查詢被拒
  - description: 驗證以無效 / 偽造 / 過期 token 呼叫 `getPerm`，後端 `getAndVerifyClientUser` 反查不到對應 client 使用者，core reject，`pm2resolve` 將 `state` 設為 `'error'`；前端 `getPerm.mjs` 偵測 `state !== 'success'` 後 reject。屬 token 守門失敗案例，與 E2E-001 共同覆蓋成敗分支。
  - flow:
    - 測試資料：一組不對應任何有效 client 使用者的 token 字串。
    - 操作：以該無效 token 呼叫 `getPerm(url, tokenTar)`。
    - 驗證：
      1. 語意：`getPerm` 回傳的 Promise 被 reject，reject 值為 `cannotGetUserDataByUrl`（後端 `state='error'` 觸發）；未取得 `user` / `rules`。
      2. 視覺：純 API 無 UI 終態，無 pixel baseline。
    - 雙語：API 流程，僅跑一輪。
    - 清理：本案被拒未寫 DB，無需清理。

- **E2E-004**
  - title: url 缺少 token 佔位符時前端即拒
  - description: 驗證呼叫 `getPerm` 時若傳入的 url 不含 `token={token}` 佔位符，`getPerm.mjs` 在送出 axios 請求「之前」即 reject，不打後端。屬前端參數前置檢查攔截案例。
  - flow:
    - 測試資料：一個合法但不含 `token={token}` 的 url（如 `http://host/api/getPerm`）與任意 tokenTar。
    - 操作：以該 url 呼叫 `getPerm(url, tokenTar)`。
    - 驗證：
      1. 語意：`getPerm` 回傳的 Promise 被 reject，reject 值為 `noTokenInUrl`；無後端請求發出。
      2. 視覺：純 API 無 UI 終態，無 pixel baseline。
    - 雙語：API 流程，僅跑一輪。
    - 清理：本案前端即拒，無需清理。

- **E2E-005**
  - title: 提供 funConvertPerm 時回傳值經轉換
  - description: 驗證呼叫 `getPerm` 時於 `opt.funConvertPerm` 傳入轉換函式，後端回傳的 `{ user, rules }` 物件會先經該函式（同步或回 Promise 皆支援）轉換後才回傳；轉換後仍須為有效物件，否則 reject。屬回傳值客製轉換案例。
  - flow:
    - 測試資料：base seed client 使用者與有效 token；一個將回傳物件加上自訂欄位（仍為有效物件）的 `funConvertPerm`。
    - 操作：以 `getPerm(url, tokenTar, { funConvertPerm })` 呼叫。
    - 驗證：
      1. 語意：回傳值為經 `funConvertPerm` 轉換後的物件（含自訂欄位）；若 `funConvertPerm` 回非物件，`getPerm` reject、值為 `noUserDataAfterConvert`。
      2. 視覺：純 API 無 UI 終態，無 pixel baseline。
    - 雙語：API 流程，僅跑一輪。
    - 清理：本案唯讀，不改 DB，無需清理。

- **E2E-006**
  - title: url 或 tokenTar 為無效引數時前端前置檢查即拒
  - description: 驗證呼叫 `getPerm` 時若傳入的 url 非字串（如空字串）或 tokenTar 非字串（如空字串），`getPerm.mjs` 在替換佔位符與送出 axios 請求「之前」的最前置參數型別檢查即 reject，分別回 `invalidUrl`（`getPerm.mjs:15`）/ `invalidTokenTar`（`getPerm.mjs:18`），不打後端。屬前端引數型別前置攔截案例，守門點早於 E2E-004 的佔位符檢查（型別檢查在替換佔位符之前）。
  - flow:
    - 測試資料：一個空字串 url（搭有效 tokenTar）與一個空字串 tokenTar（搭合法含佔位符 url）兩組無效引數。
    - 操作：分別以 `getPerm('', tokenTar)` 與 `getPerm(url, '')` 呼叫。
    - 驗證：
      1. 語意：空 url 組 `getPerm` reject、值為 `invalidUrl`；空 tokenTar 組 reject、值為 `invalidTokenTar`；皆無後端請求發出。
      2. 視覺：純 API 無 UI 終態，無 pixel baseline。
    - 雙語：API 流程，僅跑一輪。
    - 清理：本案前端即拒，無需清理。

## 回傳結構邊界

| 概念 | 內容 | 來源 |
|---|---|---|
| 後端 HTTP body | `{ state, msg }`：`state` ∈ `'success'` / `'error'` / `'cancelled'`；成功時 `msg` = `{ user, rules }`，失敗時 `msg` = 錯誤 key | `pm2resolve` 統一包裝（`WWebPerm.mjs:879`） |
| `user` | `{ ...userFind, grupsNames }`：原使用者欄位加上 `grupsNames`（僅 `isActive='y'` 群組名以 `;` 串接） | `getGenUserAndRulesByUserId`（`WWebPerm.mjs:712-716`） |
| `rules` | 規則陣列，各元素含 `name`、`isActive`（`'y'` / `'n'`）等 | `getUserRules` 計算結果（`WWebPerm.mjs:718`） |
| `getPerm` 回傳 | 成功回 `ur`（即 `msg` 物件，或經 `funConvertPerm` 轉換後物件）；失敗回 reject key | `src/getPerm.mjs:82` |
| `perm()` 快取 | `conn` 成功後將 `pmd` 拆為 `_user`（`pmd.user`）/ `_rules`（`pmd.rules`）/ `_kpRule`（`{ ruleName: isActive }`），供 `getUser` / `getRules` / `active` 讀取 | `src/perm.mjs:46-99` |
| `active(key)` | 讀 `_kpRule[key]`，值為 `'y'` 才回 `'y'`，其餘（含 key 不存在）回 `'n'` | `src/perm.mjs:117-121` |

## i18n 訊息粒度規則

本流程為 server-side 查詢 API，回傳 `{ state, msg }` 為**機器可讀值**（`state` ∈ `'success'` / `'error'` / `'cancelled'`，成功時 `msg` = `{ user, rules }` 物件、失敗時 `msg` 為 camelCase 錯誤 key）。後端 reject key（如 `tokenNoPermission` / `userNoPermission` / `cannotFindUserFromToken` / `cannotGetUserRules`）於 `server/procLang.mjs` 皆有 eng + cht 定義（`procLang.mjs:167,179,203,207`）；前端 SDK 自身 reject 的 key（如 `noTokenInUrl` / `cannotGetUserByUrl` / `cannotGetUserDataByUrl` / `noUserDataByUrl` / `noUserDataAfterConvert`）亦於 `procLang.mjs` 有 eng + cht 定義（`procLang.mjs:865,873,877,881,885`）。由呼叫端應用以 key 查 lang 決定如何呈現。故本流程訊息粒度即「一個 reject key 對應一條 procLang 雙語訊息」。

## 參數來源

| 概念 | 來源 | 流程影響 |
|---|---|---|
| `url` | 呼叫端傳入，須含 `token={token}` 佔位符 | `getPerm.mjs` 缺佔位符即前端 reject（`getPerm.mjs:25`）；`{token}` 由 `tokenTar` 取代後送 axios |
| `tokenTar` | 呼叫端傳入的當前使用者 token | 替換進 url 作為 `?token=` query；後端據此於 `getAndVerifyClientUser` 反查發起者使用者 |
| `opt.funConvertPerm` | 呼叫端可選傳入的轉換函式 | 後端回傳物件先經此函式轉換後才回傳；轉換後須為有效物件（`getPerm.mjs:66-80`） |
| token → 使用者 | 後端外部 `getUserByToken` + perm `users` 表（依 `mappingBy`，預設 `email`）反查正規化（`WWebPerm.mjs:413,431,443`） | 解出 `userIdSelf` 作為 `getGenUserAndRulesByUserId` 查權限的依據 |
| `verifyClientUser` | 部署方提供的識別函式 | 判定 token 對應使用者是否為合法瀏覽（client）使用者；不通過即 reject（`WWebPerm.mjs:491-500`） |
| 群組 / 權限 / 目標 | perm `grups` / `pemis` / `targets` 三表 | 由 `getUserRules` 合併計算出 `rules` 與有效 `grupsNames` |

## spec 規則摘要

- **權限**：`/api/getPerm` 守門點為 handler `core()` 內以 `req.query.token` 經 `getAndVerifyClientUser`（`getUserByToken` 反查 + perm `users` 表正規化 + `verifyClientUser` 部署方識別）；token 無效 / 非 client 使用者 / 查無使用者一律 reject，`state` 回 `'error'`。**僅限 client（瀏覽）使用者，不接受 app 使用者**（`WWebPerm.mjs:834`）。
- **查詢對象**：本端點固定查「token 自身所屬」使用者權限（`userSelf` → `userIdSelf`），呼叫端不可指定他人 userId。
- **回傳契約**：後端一律以 `pm2resolve` 回 `{ state, msg }`；前端 `getPerm.mjs` 以 `state === 'success'` 判成敗，成功取 `msg` 為 `{ user, rules }`，失敗 reject。
- **邊界**：
  - `url` 必含 `token={token}` 佔位符，否則 `getPerm` 在送請求前即 reject（前端前置檢查）。
  - `funConvertPerm` 為可選；提供時回傳物件先經其轉換，轉換結果須為有效物件才 resolve。
  - `grupsNames` 僅納入 `isActive='y'` 的群組名；`rules` 內各規則自帶 `isActive`，`perm().active(key)` 以此回 `'y'` / `'n'`。
  - `getUserRules` 的規則合併計算細節屬權限規則合併計算流程，本文件不展開。
- **契約**：前端 `getPerm` 以 `errTemp` 旗標短路 axios 連線層錯誤；後端 `core()` 經 `pm2resolve` 將 resolve / reject 統一轉為 `{ state, msg }` 供前端判讀。`perm().conn` 為同源 axios 取得後拆快取（`_user` / `_rules` / `_kpRule`）的薄封裝，成功 `permSuccess` / 失敗 `permError` 回呼可選。
