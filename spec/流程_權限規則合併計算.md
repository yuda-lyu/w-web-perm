# 權限規則合併計算流程

## 觸發

後端 `getGenUserAndRulesByUserId(userId)` 在組裝「使用者連同其有效權限」時，先以 `getGenUserByUserId` 取得單一使用者列（含 `cgrups` 群組關聯字串），再由 `woItems.grups/pemis/targets.select()` 一次取出全部群組（`grups`）、權限（`pemis`）、目標節點（`targets`）三張表，接著呼叫 `getUserRules(user, grups, pemis, targets)` 計算該使用者對每一個 `target` 的最終授權（`'y'` / `'n'`）（`WWebPerm.mjs:779,782,786,790,794,798`）。

`getUserRules` 是純函式（無 I/O、無副作用），輸入四個參數、輸出 `{ grups, rules }`：`grups` 為「該使用者實際生效（active 且名稱可對應）的群組」、`rules` 為「對 `targets` 中每一個節點的最終 `isActive` 判定陣列」（`mShare.mjs:202,506-509`）。本流程描述此函式內部的解析、過濾、展開、OR/AND 合併、補齊預設值的完整演算法。

由於 `getUserRules` 無 UI 終態（純單元層級計算），本流程的重要案例皆為**單元層級 case**：斷言對象為 `getUserRules` 回傳的 `{ grups, rules }` 具體值，**無視覺 baseline**。

## 重要流程

> 以下案例為單元層級驗證（無 UI 終態、無視覺 baseline），斷言對象為 `getUserRules` 回傳的 `rules`（對某 `target` 的 `isActive`）與 `grups`（生效群組清單）。資料以「user.cgrups（群組關聯）→ grups[].cpemis（群組含哪些權限）→ pemis[].crules（權限對各 target 的 y/n）→ targets（目標節點清單）」四層組成。

- **E2E-001**
  - title: 單一群組單一權限單一目標授權 y 時最終為 y
  - description: 單元層級。給定一個 target A、一個權限 P1 對 A 授權 `'y'`、一個群組 G1 以 OR 納入 P1、使用者以 OR 納入 G1，最終 A 的判定應為 `'y'`；同形但 P1 對 A 為 `'n'` 時最終為 `'n'`。驗證最基本的單鏈路 y/n 直通不被任何合併或展開干擾。
  - flow:
    - 測試資料：targets=[A]；pemis=[P1(A=y)]；grups=[G1(P1:OR)]；user.cgrups={G1:OR}。另備一組 P1(A=n) 對照。
    - 操作：呼叫 getUserRules(user, grups, pemis, targets)。
    - 驗證：
      1. 語意：回傳 rules 中 A 的 isActive 等於 `'y'`（對照組為 `'n'`）。
      2. 視覺：無（單元層級，無 UI 終態）。
    - 雙語：不適用（後端純計算，無 i18n 文字輸出）。
    - 清理：純函式無副作用，無需清理。

- **E2E-002**
  - title: 未被任何權限提及的目標一律預設為 n
  - description: 單元層級。給定多個 target 但權限僅明確涵蓋其中部分節點，未被涵蓋的 target 在最終 rules 中應一律補為 `'n'`，且回傳 rules 須涵蓋 targets 中每一個節點（不漏項）。驗證「未授權即拒絕」的補齊預設語意。
  - flow:
    - 測試資料：targets=[A,B,C,D]；pemis=[P1(A=y,B=n),P2(C=y)]；grups=[G1(P1:OR,P2:OR)]；user.cgrups={G1:OR}。
    - 操作：呼叫 getUserRules，取回 rules。
    - 驗證：
      1. 語意：A=`'y'`、C=`'y'`；B=`'n'`、D=`'n'`（D 從未被任何權限提及，補為 n）；rules 含 targets 全部節點。
      2. 視覺：無。
    - 雙語：不適用。
    - 清理：無。

- **E2E-003**
  - title: OR 於權限層級對同一目標取聯集
  - description: 單元層級。同一群組內多個 OR 權限，對同一 target 只要任一權限為 `'y'` 結果即為 `'y'`（全為 `'n'` 才 `'n'`）；對互不重疊的 target 則取聯集。驗證權限層級 OR 的「任一即可」聯集語意。
  - flow:
    - 測試資料：targets=[A,B]；pemis=[P1(A=y),P2(A=n)]（同 target 一 y 一 n）與 pemis=[P1(A=y),P2(B=y)]（不重疊）兩組；grups=[G1(P1:OR,P2:OR)]；user.cgrups={G1:OR}。
    - 操作：分別呼叫 getUserRules。
    - 驗證：
      1. 語意：同 target 組 A=`'y'`；不重疊組 A=`'y'` 且 B=`'y'`。
      2. 視覺：無。
    - 雙語：不適用。
    - 清理：無。

- **E2E-004**
  - title: OR 於群組層級對同一目標取聯集
  - description: 單元層級。使用者以 OR 納入多個群組，各群組產出的權限在群組間以 OR 聯集：任一群組授權某 target 即為 `'y'`，互不重疊的 target 取聯集。驗證 OR 聯集語意在「群組層級」與「權限層級」一致。
  - flow:
    - 測試資料：targets=[A,B]；pemis=[P1(A=y),P2(B=y)]；grups=[G1(P1:OR),G2(P2:OR)]；user.cgrups={G1:OR,G2:OR}。另備 P1(A=y)/P2(A=n) 同 target 對照。
    - 操作：呼叫 getUserRules。
    - 驗證：
      1. 語意：不重疊組 A=`'y'` 且 B=`'y'`；同 target 對照組 A=`'y'`。
      2. 視覺：無。
    - 雙語：不適用。
    - 清理：無。

- **E2E-005**
  - title: AND 以第一筆為基底交集且 obj2 獨有 key 強制為 n
  - description: 單元層級。AND 合併具非對稱性：以第一個物件（obj1）為基底，逐一與後續物件交集——只在「obj1 與 obj2 皆 y」時保留 `'y'`；只存在於 obj1 的 key 保留 obj1 原值（不受影響），只存在於 obj2（obj1 未涵蓋）的 key 一律強制為 `'n'`。驗證 AND 的「皆須授權」交集與非對稱補項語意。
  - flow:
    - 測試資料：targets=[A,B]；obj1 獨有組 pemis=[P1(A=y,B=y),P2(A=y)]、obj2 獨有組 pemis=[P1(A=y),P2(A=y,B=y)]；grups=[G1(P1:AND,P2:AND)]；user.cgrups={G1:OR}。另備 P1(A=y)/P2(A=n) 一 y 一 n 對照。
    - 操作：呼叫 getUserRules。
    - 驗證：
      1. 語意：obj1 獨有組 A=`'y'`、B=`'y'`（B 保留自基底 P1）；obj2 獨有組 A=`'y'`、B=`'n'`（B 不在基底 P1，強制 n）；一 y 一 n 對照組 A=`'n'`。
      2. 視覺：無。
    - 雙語：不適用。
    - 清理：無。

- **E2E-006**
  - title: 混合 OR AND 時先聯集再交集收緊
  - description: 單元層級。同一群組或同一使用者同時含 OR 與 AND 項時，先對所有 OR 項聯集得初步結果，再以 AND 項逐一交集收緊——AND 可把 OR 得到的 `'y'` 收緊為 `'n'`，但無法把 OR 得到的 `'n'` 反轉為 `'y'`。權限層級與群組層級皆遵循此「OR 先、AND 後」順序。
  - flow:
    - 測試資料：權限層級 targets=[A,B]、pemis=[P1(A=y,B=y),P2(A=y,B=n)]、grups=[G1(P1:OR,P2:AND)]、user={G1:OR}；群組層級 targets=[A,B,C]、pemis=[P1(A=y,B=n,C=n),P2(A=n,B=y,C=n),P3(A=y,B=y,C=y)]、grups=[G1(P1:OR),G2(P2:OR),G3(P3:OR)]、user={G1:OR,G2:OR,G3:AND}。
    - 操作：分別呼叫 getUserRules。
    - 驗證：
      1. 語意：權限層級 A=`'y'`、B=`'n'`（AND 收緊）；群組層級 A=`'y'`、B=`'y'`、C=`'n'`（OR 得 C=n，AND 無法反轉為 y）。
      2. 視覺：無。
    - 雙語：不適用。
    - 清理：無。

- **E2E-007**
  - title: isActive 為 n 的群組或權限被排除於計算之外
  - description: 單元層級。使用者關聯中 `isActive='n'` 的群組整組被忽略；群組內 `isActive='n'` 的權限亦被忽略，不參與任何 OR/AND 合併。整組或整批被忽略後仍須對所有 target 補齊預設 `'n'`，不得拋錯。驗證停用旗標的過濾語意。
  - flow:
    - 測試資料：群組停用 targets=[A]、pemis=[P1(A=y)]、grups=[G1(P1:OR)]、user={G1:{mode:OR,isActive:n}}；權限停用 targets=[A,B]、pemis=[P1(A=y,B=n),P2(A=n,B=y)]、grups=[G1(P1:OR,P2:OR_OFF)]、user={G1:OR}。
    - 操作：分別呼叫 getUserRules。
    - 驗證：
      1. 語意：群組停用組 A=`'n'`（整組忽略）；權限停用組 A=`'y'`、B=`'n'`（僅 P1 生效，P2 被忽略）；回傳 grups 僅含 active 且名稱可對應的群組。
      2. 視覺：無。
    - 雙語：不適用。
    - 清理：無。

- **E2E-008**
  - title: 引用不存在的群組或權限名稱被容錯忽略
  - description: 單元層級。使用者 `cgrups` 引用一個在 `grups` 表中找不到的群組名稱、或群組 `cpemis` 引用一個在 `pemis` 表中找不到的權限名稱時，找不到的項目被靜默忽略而非拋錯；同一容器內可對應的項目仍正常生效。驗證名稱比對的容錯（部分匹配仍生效）。
  - flow:
    - 測試資料：不存在群組 targets=[A]、user={G_NOT_EXIST:OR}；不存在權限 grups=[G1({P_NOT_EXIST:OR})]、user={G1:OR}；部分匹配 grups=[G1({P1:OR,P_NOT_EXIST:OR})]、targets=[A,B]、pemis=[P1(A=y)]。
    - 操作：分別呼叫 getUserRules。
    - 驗證：
      1. 語意：不存在群組組 A=`'n'`；不存在權限組 A=`'n'`；部分匹配組 A=`'y'`、B=`'n'`（僅 P1 生效）；過程不拋錯。
      2. 視覺：無。
    - 雙語：不適用。
    - 清理：無。

- **E2E-009**
  - title: cgrups 採 JSON5 或為空字串時容錯解析不拋錯
  - description: 單元層級。`user.cgrups`、`cpemis`、`crules` 皆以字串儲存並經 JSON5 解析，故允許單引號等 JSON5 寬鬆語法正常解析；`cgrups` 為空字串時解析降級為空容器、不拋錯，並對所有 target 補齊 `'n'`。驗證字串解析層的格式容錯與空輸入韌性。
  - flow:
    - 測試資料：JSON5 組 targets=[A]、pemis=[P1(A=y)]、grups=[G1(P1:OR)]、user.cgrups 為單引號字串 `{ 'G1': { 'mode': 'OR', 'isActive': 'y' } }`；空字串組 user.cgrups=''、grups=[]、pemis=[]、targets=[A]。
    - 操作：分別呼叫 getUserRules。
    - 驗證：
      1. 語意：JSON5 組 A=`'y'`（單引號正常解析）；空字串組 A=`'n'`（不拋錯、補預設 n）。
      2. 視覺：無。
    - 雙語：不適用。
    - 清理：無。

- **E2E-010**
  - title: 複雜權限樹兩層巢狀合併產出正確最終授權
  - description: 單元層級。多群組、多權限、巢狀路徑式 target（如 `專案A/頁A/按鈕1`）與大量稀疏目標的複雜情境，須通過「群組內合併 → 群組間合併」兩層 mergeRules，且未涵蓋的大量目標全補 `'n'`、回傳 rules 涵蓋每一個 target。驗證演算法在真實規模樹下的整體正確性。
  - flow:
    - 測試資料：路徑樹 targets=[專案A/頁A, 專案A/頁B, 專案A/頁A/按鈕1, 專案B/頁A]、對應 pemis（閱覽權 / 執行權 / 專案B全權）、grups（基本瀏覽組 / 操作員組 / 專案B組）、user={操作員組:OR, 專案B組:OR}；另備 50 個 target、僅授權前 3 個的稀疏組。
    - 操作：呼叫 getUserRules。
    - 驗證：
      1. 語意：路徑樹組 專案A/頁A=`'y'`、專案A/頁B=`'y'`、專案A/頁A/按鈕1=`'y'`（執行權 OR 閱覽權 n = y）、專案B/頁A=`'y'`；稀疏組前 3 個為 `'y'`、其餘 47 個與 target-49 皆 `'n'`，且 50 個 target 全數出現於 rules。
      2. 視覺：無。
    - 雙語：不適用。
    - 清理：無。

## 合併語意規則

| 概念 | 規則 | 對應位置 |
|---|---|---|
| OR 聯集（objOR） | 對同一 key，只要任一物件為 `'y'` 即 `'y'`；皆 `'n'`（或缺）才 `'n'`。兩物件的 key 取聯集 | `mShare.mjs:308-325` |
| AND 交集（objAND） | 以第一物件（obj1）`cloneDeep` 為基底；遍歷 obj2，只在 obj1 該 key `'y'` 且 obj2 該 key `'y'` 時保留 `'y'`，否則 `'n'` | `mShare.mjs:327-344` |
| AND 非對稱性 | obj1 獨有的 key：obj2 不遍歷到，保留 obj1 原值；obj2 獨有（obj1 未涵蓋）的 key：基底缺此 key，先補 `'n'` 再判定，等同強制 `'n'` | `mShare.mjs:331,335-340` |
| OR 先 AND 後 | mergeRules 先以全部 OR 項聯集出 `kpRuleFin`，再以全部 AND 項逐一交集收緊；故 AND 只能把 OR 得到的 `'y'` 收緊為 `'n'`，無法把 `'n'` 反轉為 `'y'` | `mShare.mjs:362-375` |
| 初值選取 | 有 OR 項時 `kpRuleFin` 起始於第一筆 OR 的 kpRule；無 OR 但有 AND 時起始於第一筆 AND 的 kpRule | `mShare.mjs:353-358` |
| 兩層套用 | 同一 mergeRules 套用兩次：群組內（合併各權限）與群組間（合併各群組結果），語意一致 | `mShare.mjs:457,473` |

## isActive 過濾規則

| 層級 | 過濾規則 | 對應位置 |
|---|---|---|
| 群組 | 使用者關聯中 `isActive='n'` 的群組整組被 `filter` 移除，不參與後續任何展開與合併 | `mShare.mjs:404` |
| 權限 | 群組 `cpemis` 中 `isActive='n'` 的權限被 `filter` 移除，不納入該群組的合併 | `mShare.mjs:411` |
| 合併效果 | 過濾在「分離 OR/AND 之前」執行，故被停用者完全不出現於 `ts`；若整批被過濾為空，`mergeRules` 回傳空陣列、最終由 targets 補齊全 `'n'` | `mShare.mjs:304-305,488-493` |
| mode 篩選 | 即使 isActive='y'，`mergeRules` 仍會跳過 `mode` 非 `'OR'` / `'AND'` 的項目（不計入合併） | `mShare.mjs:284-285` |

## 名稱匹配與容錯規則

| 情境 | 行為 | 對應位置 |
|---|---|---|
| 群組名比對 | `arrFinder` 以 `name` 將 `cgrups` 項與 `grups` 表逐一比對，找不到對應者不加入結果（靜默忽略） | `mShare.mjs:400,256-272` |
| 權限名比對 | `arrFinder` 以 `name` 將 `cpemis` 項與 `pemis` 表比對，找不到者忽略 | `mShare.mjs:412` |
| 部分匹配 | 同一容器內，可對應的項目正常生效，僅找不到的項目被略過 | `mShare.mjs:256-272` |
| JSON5 解析 | `cvStrToArr` 以 `JSON5.parse` 解析，允許單引號等寬鬆語法；解析失敗時 `console.log` 後降級為空容器、不向外拋錯 | `mShare.mjs:232-239` |

## spec 規則摘要

- **資料層級**：權限樹分四層——使用者（`cgrups`）→ 群組（`cpemis`）→ 權限（`crules`）→ 目標（`targets`）；每層字串以 JSON5 解析，逐層展開後做兩次合併（群組內、群組間）。
- **合併語意**：
  - **OR**：對同一 target 取聯集，任一為 `'y'` 即 `'y'`。
  - **AND**：以第一筆為基底取交集，皆 `'y'` 才 `'y'`；具非對稱性——obj1 獨有 key 保留原值、obj2 獨有（基底未涵蓋）key 強制 `'n'`。
  - **順序**：固定「OR 先聯集 → AND 後交集」，AND 只能收緊、不能放寬（無法把 OR 得到的 `'n'` 反轉為 `'y'`）。
- **過濾**：`isActive='n'` 的群組與權限在分離 OR/AND 之前即被 `filter` 移除，不參與合併；`mode` 非 `'OR'` / `'AND'` 的項目於 `mergeRules` 內被跳過。
- **容錯**：名稱比對找不到的群組 / 權限被靜默忽略（部分匹配仍生效）；JSON5 解析失敗或空字串降級為空容器、不拋錯。
- **預設**：所有未被任何生效規則涵蓋的 `target` 一律補為 `'n'`（未授權即拒絕）；回傳 rules 必涵蓋 `targets` 全部節點。
- **契約**：`getUserRules` 為純函式，輸入 `(user, grups, pemis, targets)`、輸出 `{ grups, rules }`，無 I/O、無副作用，可逐 case 以回傳值斷言（單元層級，無 UI 終態、無視覺 baseline）。
