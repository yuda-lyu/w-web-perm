# 查詢指定使用者權限流程

## 觸發

外部 node 應用（非瀏覽器使用者）以系統介接身分查詢「某一指定 `userId`」的權限資訊。應用端呼叫 `getPermUserInfor(url, tokenSelf, userIdTar, opt)`（`src/getPermUserInfor.mjs:9`），其中 `url` 帶有 `token={sysToken}` 與 `userId={userId}` 兩個佔位符（`src/getPermUserInfor.mjs:10`），函式以 `tokenSelf` 置換 `{sysToken}`、以 `userIdTar` 置換 `{userId}`（`src/getPermUserInfor.mjs:31-32`）後，以 `axios.get` 打後端 `GET /api/getPermUserInfor`（`server/WWebPerm.mjs:888-962`）。

後端以 `query.token` 解出發起者 `tokenSelf`，經 `getAndVerifyAppUser` 驗證其為合法 app 使用者（`server/WWebPerm.mjs:910`）；驗證通過後以 `query.userId` 為查找對象，呼叫 `getGenUserAndRulesByUserId` 取回該使用者的權限資訊物件並回傳（`server/WWebPerm.mjs:936`）。`getAndVerifyAppUser` 僅以外部 `getUserByToken` 取得使用者再過 `verifyAppUser` 把關（`server/WWebPerm.mjs:514,529`），**不反查 perm `users` 表**，故可支援 `getUserByToken` 針對應用程式所給予之虛擬使用者帳號（`server/WWebPerm.mjs:509`）。

## 重要流程

本流程為系統介接 API，無瀏覽器 UI 終態，E2E 以 API 回傳值斷言為主，不含視覺 baseline。

- **E2E-001**
  - title: 合法 app token 查詢既有 userId 回傳該使用者權限
  - description: 驗證以合法 app 使用者 token（`tokenSelf`）查詢一個存在且 `isActive='y'` 的 `userId` 時，後端通過 `getAndVerifyAppUser` 驗證、`getGenUserAndRulesByUserId` 取回資料，HTTP 回應 `state='success'`、`msg` 為含 `user` 與 `rules` 的物件；客戶端函式 resolve 該物件。屬查詢成功 golden 案例。
  - flow:
    - 測試資料：base seed 既有 user（取一筆 `isActive='y'` 的 `userId` 作查找對象）；發起者 token 為部署方 `getUserByToken` 可解出且通過 `verifyAppUser` 的 app token。
    - 操作：以合法 `tokenSelf` 與既有 `userIdTar` 呼叫 `getPermUserInfor(url, tokenSelf, userIdTar)`。
    - 驗證：
      1. 語意：HTTP 回應 `state='success'`；`msg` 為物件，含 `user`（其 `id` 等於查找的 `userIdTar`，且帶 `grupsNames` 字串欄位）與 `rules`（陣列）；客戶端函式 resolve 該物件。
      2. 視覺：本流程為 API 查詢、無 UI 終態，無視覺 baseline。
    - 雙語：本流程為 API，回傳為資料物件、不含 i18n 文案，單輪即可。
    - 清理：本案唯讀，不改 DB，無需清理。

- **E2E-002**
  - title: 缺少 token 佔位符的 url 在客戶端前置攔截
  - description: 驗證傳入的 `url` 同時缺少 `token={sysToken}` 與 `userId={userId}` 佔位符時，客戶端函式於送出請求前即 reject `noTokenUserIdInUrl`，不打後端。屬客戶端前置參數攔截案例。
  - flow:
    - 測試資料：建構一個不含任一佔位符的 `url`；`tokenSelf` / `userIdTar` 給合法非空字串。
    - 操作：以該 `url` 呼叫 `getPermUserInfor`。
    - 驗證：
      1. 語意：函式 reject 值 `noTokenUserIdInUrl`；後端未收到請求。
      2. 視覺：無 UI 終態，無視覺 baseline。
    - 雙語：API 流程，reject 值為 camelCase key（`procLang.mjs` 有 eng + cht 定義），單輪即可。
    - 清理：未送出請求、未改 DB，無需清理。

- **E2E-003**
  - title: 缺漏必填參數在客戶端前置攔截
  - description: 驗證 `url` / `tokenSelf` / `userIdTar` 任一為非有效字串時，客戶端函式於對應檢查點 reject `invalidUrl` / `invalidTokenSelf` / `invalidUserIdTar`，不打後端。屬客戶端必填檢查案例。
  - flow:
    - 測試資料：分別建構三組輸入，各令 `url` / `tokenSelf` / `userIdTar` 其一為空字串、其餘給合法值。
    - 操作：以各組輸入呼叫 `getPermUserInfor`。
    - 驗證：
      1. 語意：依缺漏欄位分別 reject `invalidUrl` / `invalidTokenSelf` / `invalidUserIdTar`；後端未收到請求。
      2. 視覺：無 UI 終態，無視覺 baseline。
    - 雙語：API 流程，reject 值為 camelCase key（`procLang.mjs` 有 eng + cht 定義），單輪即可。
    - 清理：未送出請求、未改 DB，無需清理。

- **E2E-004**
  - title: token 非合法 app 使用者後端拒絕
  - description: 驗證 `tokenSelf` 無法經 `getUserByToken` 解出使用者、或解出之使用者未通過 `verifyAppUser` 時，後端 `getAndVerifyAppUser` reject，HTTP 回應 `state='error'`；客戶端因 `state!=='success'` 而 reject `cannotGetUserDataByUrl`。屬後端 app 身分驗證拒絕案例。
  - flow:
    - 測試資料：base seed；`tokenSelf` 給一個部署方 `getUserByToken` 解不出使用者（或解出但 `verifyAppUser` 回 false）的 token；`userIdTar` 給既有 userId。
    - 操作：以該 `tokenSelf` 呼叫 `getPermUserInfor`。
    - 驗證：
      1. 語意：HTTP 回應 `state='error'`（`msg` 為 `cannotFindUserFromToken`（getUserByToken 解不出使用者）或 `userNoPermission`（verifyAppUser 回 false））；客戶端函式 reject `cannotGetUserDataByUrl`；DB 不變。
      2. 視覺：無 UI 終態，無視覺 baseline。
    - 雙語：API 流程，回傳 reject 字串、不含 i18n，單輪即可。
    - 清理：本案唯讀，不改 DB，無需清理。

- **E2E-005**
  - title: 查詢不存在的 userId 回傳空權限（全停用，fail-safe）
  - description: 驗證 `tokenSelf` 為合法 app 使用者、但 `userIdTar` 對應之使用者不存在時，後端 `getGenUserByUserId` 取不到使用者，但 `getUserRules` 容忍此情形、回傳「全停用」之空權限物件（fail-safe），故 HTTP 回應 `state='success'`、`msg` 為一個 `user` 無有效 `id`、`rules` 各規則 `isActive='n'` 的物件；客戶端 resolve 該物件。屬「查找對象不存在 → 回空權限」之 fail-safe 案例（**非 reject**）。
  - flow:
    - 測試資料：合法 `tokenSelf`；`userIdTar` 給一個 `users` 表中不存在的 id。
    - 操作：以該輸入呼叫 `getPermUserInfor`。
    - 驗證：
      1. 語意：HTTP 回應 `state='success'`；客戶端 resolve 之物件 `user` 無有效 `id`、`rules` 為陣列且各規則 `isActive='n'`（空權限）；DB 不變。
      2. 視覺：無 UI 終態，無視覺 baseline。
    - 雙語：API 流程，回傳為資料物件、不含 i18n，單輪即可。
    - 清理：本案唯讀，不改 DB，無需清理。

- **E2E-006**
  - title: funConvertPerm 轉換查詢結果
  - description: 驗證 `opt.funConvertPerm` 提供時，客戶端於取得後端回傳的權限物件後，以 `funConvertPerm` 轉換（支援同步或回傳 Promise）並改以轉換後物件 resolve；若轉換結果非有效物件則 reject `noUserDataAfterConvert`。屬客戶端後處理 hook 案例。
  - flow:
    - 測試資料：同 E2E-001 之合法 `tokenSelf` 與既有 `userIdTar`；`opt.funConvertPerm` 給一個將原物件包裝/改寫的函式。
    - 操作：以帶 `funConvertPerm` 的 `opt` 呼叫 `getPermUserInfor`。
    - 驗證：
      1. 語意：函式 resolve 之物件為 `funConvertPerm` 轉換後的結果（與未轉換之原物件不同）；當 `funConvertPerm` 回非物件時改 reject `noUserDataAfterConvert`。
      2. 視覺：無 UI 終態，無視覺 baseline。
    - 雙語：API 流程，回傳為資料物件、不含 i18n，單輪即可。
    - 清理：本案唯讀，不改 DB，無需清理。

## i18n 訊息粒度規則

本流程為 server-side 查詢 API，回傳 `{ state, msg }` 為**機器可讀值**（`state='success'` / `'error'`，成功時 `msg` 為含 `user` / `rules` 的權限物件、失敗時 `msg` 為 camelCase 錯誤 key）。後端 reject key（如 `tokenNoPermission` / `cannotFindUserFromToken` / `userNoPermission` / `userIdNoPermrules` / `cannotGetUserRules`）於 `server/procLang.mjs` 皆有 eng + cht 定義（`procLang.mjs:207,167,179,211,203`）；前端 SDK 自身 reject 的 key（如 `noTokenUserIdInUrl` / `cannotGetUserByUrl` / `cannotGetUserDataByUrl` / `noUserDataByUrl` / `noUserDataAfterConvert`）亦於 `procLang.mjs` 有 eng + cht 定義（`procLang.mjs:869,873,877,881,885`）。由呼叫端應用以 key 查 lang 決定如何呈現。故本流程訊息粒度即「一個 reject key 對應一條 procLang 雙語訊息」。

## spec 規則摘要

- **權限**：發起者身分以 `query.token`（即 `tokenSelf`）經 handler 內 `getAndVerifyAppUser` 把關（`WWebPerm.mjs:910`），驗證鏈為 `getUserByToken` 取使用者 → `verifyAppUser` 確認為合法 app 使用者（`WWebPerm.mjs:514,529`）。此驗證**只依外部 `getUserByToken`、不反查 perm `users` 表**，故可支援 `getUserByToken` 給予應用程式專屬之虛擬使用者帳號（`WWebPerm.mjs:509`）。
- **查找對象**：查的是 `query.userId`（即 `userIdTar`）所指之**任意指定使用者**，由 `getGenUserAndRulesByUserId(userIdFind)` 取其權限（`WWebPerm.mjs:936`）；`getGenUserByUserId` 經 `getGenUserByKV` 僅取 `isActive='y'` 之使用者（`WWebPerm.mjs:664-667`、`651`）。**查找對象不存在時不 reject**：`getUserRules` 容忍取不到的使用者、回傳「全停用」之空權限物件（fail-safe），故 handler 回 `state='success'`、`msg` 為 `user` 無有效 `id`、`rules` 各規則 `isActive='n'` 的物件（見 E2E-005）。
- **回傳契約**：handler 以 `pm2resolve(core)` 統一輸出 `{ state, msg }`（`WWebPerm.mjs:954`）——成功 `state='success'`、`msg` 為 `{ user, rules }`；失敗 `state='error'`、`msg` 為 reject key。客戶端依 `state==='success'` 判定成敗，再取 `msg` 為權限物件（`getPermUserInfor.mjs:56,61`）。
- **客戶端前置攔截**：`url` / `tokenSelf` / `userIdTar` 任一非有效字串即於送出前 reject（`getPermUserInfor.mjs:14-22`）；`url` 缺兩個佔位符亦 reject（`getPermUserInfor.mjs:28-29`），不打後端。
- **客戶端後處理**：`opt.funConvertPerm` 為函式時，於成功回傳後對權限物件做轉換（支援同步或 Promise），轉換結果須為有效物件，否則 reject（`getPermUserInfor.mjs:70-84`）。
- **錯誤分層**：axios 連線層級失敗 → `errTemp` 非 null → reject `cannotGetUserByUrl`（`getPermUserInfor.mjs:42-43`）；後端回 `state!=='success'` → reject `cannotGetUserDataByUrl`（`getPermUserInfor.mjs:56-57`）；回傳物件非有效 → reject `noUserDataByUrl`（`getPermUserInfor.mjs:65-66`）。
