# 設計要點與取捨

本檔記錄 w-web-perm 專案之跨流程 (cross-cutting) 設計決策.
每條 ADR 一段, 含: Status / Context / Decision / Consequences / Affected.

修改決策時新增 ADR (編號遞增) 並把舊條 Status 改為 Superseded, **不刪舊條**
保留決策歷史, 方便日後 audit 追溯為什麼當初這樣選.

> 本專案核心為「權限樹解析」: 四層資料 (使用者 `cgrups` → 群組 `cpemis` → 權限 `crules` → 目標 `targets`) 逐層展開, 經 OR/AND 兩階段合併算出某使用者對每個 target 的最終 `'y'`/`'n'`. 解析主體為純函式 `getUserRules` (`src/plugins/mShare.mjs`, 前後端共用), 由後端 `getGenUserAndRulesByUserId` (`server/WWebPerm.mjs:779`) 與外部介接 API 共同消費. 以下 ADR 多圍繞此解析鏈與其周邊寫入 / 同步 / 測試守護之橫切契約.

## ADR-001: 權限合併語意 — OR 聯集 / AND 非對稱交集 / OR 先 AND 後

**Status**: Accepted (2026-06-09)

**Context**:
- 同一使用者可同時納入多個群組、每個群組可納入多個權限, 每個權限對同一 target 可能給出不同的 `'y'`/`'n'`. 須訂出明確且唯一的合併語意, 否則「使用者最終對某 target 有沒有權限」無法被確定性判定.
- 合併同時發生於兩個層級 (群組內合併各權限、群組間合併各群組結果), 兩層若採不同語意會讓權限樹難以推理.
- 不解決會踩到: 權限判定不確定、同一筆資料在不同 caller 算出不同結果、無法寫可斷言的 unit 測試.

**Decision**:
- **OR 聯集 (`objOR`)**: 對同一 key, 任一物件為 `'y'` 即 `'y'`; 皆 `'n'` (或缺) 才 `'n'`; 兩物件的 key 取聯集 (`mShare.mjs:308-325`).
- **AND 交集 (`objAND`)**: 以第一物件 (obj1) `cloneDeep` 為基底, 遍歷 obj2 — 只在 obj1 該 key 為 `'y'` 且 obj2 該 key 為 `'y'` 時保留 `'y'`, 否則 `'n'` (`mShare.mjs:327-344`).
- **AND 非對稱性 (load-bearing)**: obj1 獨有的 key (obj2 不遍歷到) 保留 obj1 原值; obj2 獨有 (基底未涵蓋) 的 key 先補 `'n'` 再判定, 等同**強制 `'n'`** (`mShare.mjs:331-340`). AND 交換 obj1/obj2 順序結果可能不同, 此非對稱性是刻意行為, 非 bug.
- **OR 先 AND 後 (固定順序)**: `mergeRules` 先取所有 OR 項聯集出 `kpRuleFin`, 再以所有 AND 項逐一交集收緊 (`mShare.mjs:346-375`). 故 AND 只能把 OR 得到的 `'y'` **收緊**為 `'n'`, 無法把 OR 得到的 `'n'` **反轉**為 `'y'`.
- **初值選取**: 有 OR 項時 `kpRuleFin` 起始於第一筆 OR 的 `kpRule`; 無 OR 但有 AND 時起始於第一筆 AND 的 `kpRule` (`mShare.mjs:351-358`).
- **同一函式套兩層**: 群組內 (合併各權限) 與群組間 (合併各群組結果) 皆呼叫同一 `mergeRules`, 兩層語意一致.

**Consequences**:
- ✅ 權限判定確定性: 同一筆四層資料永遠算出同一棵權限樹, 可逐 case 以回傳值斷言.
- ✅ 兩層共用同一合併語意, 權限樹易於推理, 不出現「群組內一套、群組間另一套」的不一致.
- ✅ 「OR 先 AND 後」使 AND 的收緊意圖明確 (AND 是限制器, 不是放寬器).
- ⚠️ AND 非對稱性違反一般人對「交集應對稱」的直覺: 對調 AND 兩項可能改變結果 (obj2 獨有 key 被強制 `'n'`). 設計取捨已由 E2E-005 (`流程_權限規則合併計算.md`) + unit 測試 (`getUserRules.test.mjs:284-317`) 凍結為契約, 後續不可「順手對稱化」.
- ⚠️ `mode` 非 `'OR'` / `'AND'` 的項目於 `mergeRules` 內被靜默跳過 (`mShare.mjs:284-285`), 不計入合併; 拼字錯誤的 mode 等於該項不存在.

**Affected**:
- src/plugins/mShare.mjs:280-389 (mergeRules / objOR / objAND)
- test/getUserRules.test.mjs:256-451, 736-877 (OR/AND/混合/兩層巢狀)
- spec/流程_權限規則合併計算.md (合併語意規則表 / E2E-001~006)

## ADR-002: 未授權即拒絕 — 未涵蓋的 target 一律補 `'n'`

**Status**: Accepted (2026-06-09)

**Context**:
- 權限只明確記載「對哪些 target 授 `'y'` / `'n'`」, 但 `targets` 全集通常遠大於任一使用者實際被提及的範圍 (稀疏授權).
- 對「從未被任何生效規則提及」的 target, 預設應為授權還是拒絕? 安全系統的預設必須是拒絕 (fail-closed).
- 不解決會踩到: 預設放寬 = 新增 target 後使用者自動取得權限的安全破口.

**Decision**:
- 合併結果填入 `kp` 後, 以**當前 `targets` 全集**逐項輸出, 每個 target 取其解析後 `isActive`, 取不到者 `get(kp, t.id, 'n')` 補為 `'n'` (`mShare.mjs:485-503`, 詳見 ADR-003 之輸出主軸).
- 回傳 `rules` 必涵蓋 `targets` 中每一個節點 (不漏項), 使消費端可對任一 target 直接查表.
- 整批被過濾為空 (見 ADR-004) 或 `cgrups` 為空時, `mergeRules` 回傳空陣列, 最終仍由 targets 補齊全 `'n'`, 不拋錯 (`mShare.mjs:304-305`).

**Consequences**:
- ✅ fail-closed: 未明確授權即拒絕, 新增 target 不會讓既有使用者自動取得權限.
- ✅ 回傳 rules 涵蓋 targets 全集, 消費端查表不需處理「查無此 target」分支.
- ⚠️ `targets` 為空陣列時回傳空 rules (`getUserRules.test.mjs:101-111`); 消費端須容忍空 rules.

**Affected**:
- src/plugins/mShare.mjs:485-503 (expand 段補 'n')
- test/getUserRules.test.mjs:119-149, 707-732 (target 展開 / 稀疏權限)
- spec/流程_權限規則合併計算.md (E2E-002 / spec 規則摘要「預設」)

## ADR-003: 權限樹以「當前 targets」為輸出主軸, 剔除孤兒 target

**Status**: Accepted (2026-06-09, commit 51bd712)

**Context**:
- 管理員刪除某 target 後, 引用該 target 的 `pemis.crules` 字串仍可能殘留對它的 `'y'`/`'n'` 引用 (crules 與 targets 為各自獨立維護的兩張表, 刪 target 不會回頭清掃所有 crules).
- 修正前 `getUserRules` 的 expand 段把最新 `targets` 只拿來「補缺」(`if (!haskey(kp, t.id)) kp[t.id]='n'`) 而**未拿來過濾**, 輸出集合 = `crules keys ∪ targets`. 結果: 已刪除的 target 因 crules 殘留引用而仍出現在解析後權限樹, `getPermUserInfor` 會回傳已刪除管控對象的授權 — **外部應用查到不該有的權限** (真實 bug, 非推測).
- 不解決會踩到: 刪 target 不生效於外部介接查詢, 權限樹與「現存管控對象」不一致.

**Decision**:
- expand 段改以「當前 `targets`」逐項為**輸出主軸**: 每個 target 取其解析後 `isActive` (取自合併 crules 的 `kp`、預設 `'n'`), `crules` 仍引用但已從 `targets` 刪除的孤兒 key 自然被濾掉 (`mShare.mjs:485-503`).
- 修正前: 輸出 = `crules keys ∪ targets`; 修正後: 輸出 = `targets` (孤兒被剔除). no-orphan 情境 (`crules ⊆ targets`) 輸出集合不變, 無退化.
- 此修正同時滿足 ADR-002「未涵蓋補 `'n'`」: 主軸換成 targets 後, 補預設與剔孤兒由同一段邏輯一次完成.

**Consequences**:
- ✅ 解析後權限樹 = 現存 targets, 刪 target 立即反映於外部介接查詢, 不再殘留孤兒授權.
- ✅ rules 筆數恆等於現存 targets 數, 成為可斷言的不變式 (E2E-005 永久回歸守護).
- ✅ no-orphan 情境輸出不變, 對既有資料無退化.
- ⚠️ 此為「以 targets 為權威主軸」的單向契約: 若未來需求要保留「crules 有引用但 targets 沒有」之孤兒於樹中 (反向), 須重新設計 — 但此需求與 fail-closed 精神相悖, 預設不支援.

**Affected**:
- src/plugins/mShare.mjs:485-503 (expand 段, commit 51bd712 改寫)
- test/e2e-targets.test.mjs (E2E-005-delete-ok 孤兒不變式語意斷言)
- spec/流程_權限規則合併計算.md (執行流程 025-027)
- spec/流程_查詢指定使用者權限.md (外部查詢消費此樹)

## ADR-004: `isActive='n'` 的群組 / 權限於合併前即被過濾

**Status**: Accepted (2026-06-09)

**Context**:
- 群組與權限皆有 `isActive` 旗標, 用於「暫時停用而不刪除」. 停用者是否應參與 OR/AND 合併?
- 若停用者仍進入合併, 一個被停用的 OR 權限仍可貢獻 `'y'`, 停用形同無效.
- 不解決會踩到: 停用旗標失去意義、管理員以為停用了卻仍生效.

**Decision**:
- **群組層級**: 使用者關聯中 `isActive='n'` 的群組整組被 `filter` 移除, 不參與後續任何展開與合併 (`mShare.mjs:404`).
- **權限層級**: 群組 `cpemis` 中 `isActive='n'` 的權限被 `filter` 移除, 不納入該群組的合併 (`mShare.mjs:411`).
- 過濾在「分離 OR/AND 之前」執行, 故被停用者完全不出現於 `ts`; 若整批被過濾為空, 由 ADR-002 補齊全 `'n'`、不拋錯.
- 回傳 `grups` 僅含 active 且名稱可對應的群組 (`mShare.mjs:506-509`), 後端據此組 `grupsNames` (`WWebPerm.mjs:809-810` 註「僅會提供 isActive='y'」).

**Consequences**:
- ✅ 停用旗標語意明確: 停用 = 完全不參與計算, 等同暫時移除.
- ✅ 過濾發生於合併前, 不會出現「停用權限仍貢獻 OR `'y'`」的洩漏.
- ✅ `grupsNames` 天然只含有效群組, 消費端不需再過濾.
- ⚠️ `isActive` 與 ADR-001 的 `mode` 篩選 (跳過非 OR/AND) 為兩道獨立過濾; 兩者皆通過才參與合併.

**Affected**:
- src/plugins/mShare.mjs:404, 411, 506-509
- server/WWebPerm.mjs:809-810 (grupsNames 取 active 群組)
- test/getUserRules.test.mjs:457-521, 603-617
- spec/流程_權限規則合併計算.md (isActive 過濾規則表)

## ADR-005: 名稱比對找不到即靜默忽略 + JSON5 容錯解析 + 空字串降級

**Status**: Accepted (2026-06-09)

**Context**:
- `cgrups` / `cpemis` / `crules` 皆以字串儲存, 經 JSON5 解析後以 `name` 跨表比對. 解析或比對可能因外部資料品質失敗 (引用了不存在的群組/權限名、字串格式寬鬆、欄位為空).
- `getUserRules` 是計算權限樹的核心純函式, 任一輸入瑕疵若直接拋錯, 會讓整個使用者的權限查詢 (含外部介接) 失敗.
- 不解決會踩到: 單筆髒資料 (例如一個已刪群組仍被引用) 導致整個權限查詢 500、所有 target 取不到判定.

**Decision**:
- **名稱比對容錯**: `arrFinder` 以 `name` 將 `cgrups` 項對 `grups` 表、`cpemis` 項對 `pemis` 表逐一比對, 找不到對應者**靜默忽略**而非拋錯 (`mShare.mjs:400, 412`); 同一容器內可對應的項目仍正常生效 (部分匹配).
- **JSON5 解析**: `cvStrToArr` 以 `JSON5.parse` 解析, 允許單引號等寬鬆語法; 解析失敗時 `console.log` 後**降級為空容器**、不向外拋錯 (`mShare.mjs:232-239`).
- **空字串 / 空容器**: `cgrups` 為空字串時解析得空容器, 後續 filter / merge 皆空, 最終靠 ADR-002 補齊全 `'n'` (`mShare.mjs:392, 488-493`).
- 此「容錯不拋錯」是 ADR-006 fail-safe 在純函式層的基礎.

**Consequences**:
- ✅ 單筆髒資料不會讓整個權限查詢失敗; 壞的部分被略過, 好的部分照常生效.
- ✅ 字串欄位允許 JSON5 寬鬆語法, 對人工編輯/外部來源較友善.
- ✅ 空輸入韌性: 空字串/空表降級為空容器, 不拋錯.
- ⚠️ 靜默忽略無使用者可見回饋: 管理員若把群組名打錯, 該群組默默不生效而無錯誤提示 — 須靠後端 `console.log` 或 e2e 守護察覺.
- ⚠️ 「找不到即忽略」使「引用了不存在的名稱」與「正確但無授權」對最終樹的效果相同 (皆 `'n'`), debug 時須查 log 區分.

**Affected**:
- src/plugins/mShare.mjs:232-239 (cvStrToArr JSON5), :256-272 (arrFinder), :392, :400, :412
- test/getUserRules.test.mjs:527-560 (名稱匹配失敗), :883-897 (JSON5 格式), :91-99 (空字串)
- spec/流程_權限規則合併計算.md (名稱匹配與容錯規則表)

## ADR-006: 查詢指定使用者權限 — 查找對象不存在回「全停用空權限」(fail-safe, 非 reject)

**Status**: Accepted (2026-06-09)

**Context**:
- 外部 node 應用以 app token 經 `GET /api/getPermUserInfor` 查詢「某一指定 `userId`」的權限. 查找對象 (`userIdTar`) 可能不存在於 perm `users` 表 (例如該 user 已被刪、或外部應用持有的是舊 userId).
- 查找對象不存在時, 後端應 reject 還是回空權限? reject 會讓外部應用須處理錯誤分支; 回空權限則統一走成功路徑.
- 不解決會踩到: 外部應用對「user 不存在」與「user 存在但無權限」須寫兩套處理、權限查詢的錯誤面變大.

**Decision**:
- `getGenUserByUserId` 取不到使用者時 (回 `null`), `getUserRules` **容忍**此情形 (`null` user 之 `cgrups` 經 ADR-005 容錯降級為空容器), 回傳「全停用」之空權限物件 — `user` 無有效 `id`、`rules` 各規則 `isActive='n'` (fail-safe).
- 故 handler 回 `state='success'`、`msg` 為含 `user`/`rules` 的物件 (`WWebPerm.mjs:1030, 812`), **非 reject** (見 `流程_查詢指定使用者權限.md` E2E-005).
- 對比: 發起者身分驗證 (app token) 失敗仍 reject 走 `state='error'` (見 ADR-007) — 「查找對象不存在」與「發起者無權限」分屬兩種路徑, 前者 fail-safe、後者守門拒絕.
- 發起者驗證**不反查 perm `users` 表**, 僅以部署方注入的 `getUserByToken` 取使用者再過 `verifyAppUser` (`WWebPerm.mjs:620, 635`), 故可支援應用程式專屬之虛擬使用者帳號.

**Consequences**:
- ✅ 外部應用對「user 不存在」與「user 無權限」統一走成功路徑取空權限樹, 處理邏輯一致.
- ✅ fail-safe 預設全停用, 對齊 ADR-002 fail-closed 精神 — 查不到 = 沒有任何權限, 不會誤放行.
- ✅ 建立於 ADR-005 純函式容錯之上, 後端不需為「user 不存在」寫額外分支.
- ⚠️ 外部應用無法單從 `state` 區分「user 不存在」與「user 存在但全 `'n'`」, 須檢視 `user.id` 是否有效 — 屬已知取捨.

**Affected**:
- server/WWebPerm.mjs:782, 798, 1030 (getGenUserByUserId / getUserRules / handler)
- src/getPermUserInfor.mjs (客戶端依 state==='success' 判定, resolve 物件)
- spec/流程_查詢指定使用者權限.md (E2E-005 / spec 規則摘要「查找對象」)

## ADR-007: 外部介接守門 — app token 經 `getAndVerifyAppUser`, 不經 verifyConn / Bearer

**Status**: Accepted (2026-06-09)

**Context**:
- `/api/getPermUserInfor` (查詢) 與 `/syncAndReplaceTabs` (同步寫入) 為直接 `server.route` 註冊的自訂端點, 供外部 node 應用以 query `token` 介接, 不走瀏覽器使用者通道.
- ORM exec 通道由 `verifyConn` 守門; 但這兩個自訂端點不經該通道, 須自帶守門, 否則任何人皆可查/改權限資料.
- 不解決會踩到: 自訂端點裸奔、外部任意 caller 可查任一 userId 權限或覆寫整表.

**Decision**:
- 兩端點守門皆由 handler 內 `getAndVerifyAppUser(token, from)` 負責 (`WWebPerm.mjs:614`): 以 `getUserByToken(token)` 取使用者 → 以部署方注入的 `verifyAppUser(user, from)` 判定為合法 app 使用者; 任一步失敗即 `Promise.reject` 走 `state='error'`.
- `verifyAppUser` 為部署方注入 `WWebPerm` 建構參數的唯一守門判定函式 (`WWebPerm.mjs:138, 635`); 信任邊界集中於此一函式.
- token 缺 / 無效 / 取不到使用者 / `verifyAppUser` 回 false 一律 reject, 回傳 `{ state:'error', msg }` (英文字串如 `token does not have permission` / `user does not have permission`), **非 i18n** (`procLang.mjs` 無對應鍵, 由呼叫端應用自行呈現).
- handler 統一以 `pm2resolve(core)` 把 core 的 resolve/reject 轉為 `{ state, msg }` (`WWebPerm.mjs:1045, 1128`).

**Consequences**:
- ✅ 自訂介接端點守門集中於 `getAndVerifyAppUser` 單一入口, 兩端點共用, 易 audit.
- ✅ 守門判定 (`verifyAppUser`) 由部署方注入, perm 本身不綁定身分系統, 介接彈性高.
- ✅ 回傳為機器可讀 `{ state, msg }`, 與瀏覽器 i18n 通道解耦.
- ⚠️ app token 等同高敏資料: 外洩即可查/改權限資料, 須限制傳遞通路.
- ⚠️ `verifyAppUser` 為部署方責任; 若注入的實作把關不嚴, 守門隨之鬆動 — perm 無法替部署方補強.

**Affected**:
- server/WWebPerm.mjs:138, 614-647 (getAndVerifyAppUser / verifyAppUser 注入)
- server/WWebPerm.mjs:985-1051 (getPermUserInfor handler), :1057-1134 (syncAndReplaceTabs handler)

## ADR-008: 外部同步採「依 `from` 全量取代」+ 空 rows 被拒 (防誤清源)

**Status**: Accepted (2026-06-09)

**Context**:
- `/syncAndReplaceTabs` 讓外部應用把本地維護的某張表 (`targets`/`pemis`/`grups`/`users` 四選一) 整批推送至伺服器, 由伺服器覆寫該應用在該表的資料. 多個外部應用各自推送, 須彼此隔離不互相覆蓋.
- 取代模型有兩種: 逐列 diff (與後台瀏覽器寫入相同, 見 ADR-009) 或依來源全量取代. 同步場景下「外部應用本地即真理」, 全量取代語意更直觀.
- 若全量取代允許空 rows, 外部應用一旦誤送 `rows=[]` 會把該來源整批清空 (資料災難).
- 不解決會踩到: 跨應用資料互相覆蓋、外部誤送空集合清空權限源.

**Decision**:
- **全量取代**: `syncAndReplaceTabs` 以 payload 的 `from` 為範圍鍵, 先 `delAll({ from })` 刪該應用既有列, 再整批 `procOrm insert` (`WWebPerm.mjs:736, 740`); **非逐列 diff** (無 `ltdtDiffByKey`、無 `ckKey` 去重). 各應用以各自 `from` 隔離.
- **空 rows 被拒**: `parsePayload` 以 `isearr` 要求 rows 為**非空**陣列 (`WWebPerm.mjs:683`); `rows=[]` 即 reject `invalid rows in payload`, **未進 delAll**, 故該 `from` 既有列不被清空. **不支援「以空集合清源」** — 此為防外部應用誤送空陣列而整批清空資料之安全閘.
- **白名單守門**: query `keyTable` 須屬 `[targets, pemis, grups, users]` 四者之一, 否則 reject (`WWebPerm.mjs:1089-1092`).
- **欄位邊界**: 端側 `provideTabs` 以白名單 `ltdtpick` 過濾欄位 (`provideTabs.mjs:8-63, 83`); 稽核欄位 (`userId` / `time*`) 一律由後端 `procOrm` 補, 不由 payload 帶入; `order` 在此路徑沿用外部帶入值, 不重給 (`WWebPerm.mjs:726` 註).

**Consequences**:
- ✅ 跨應用以 `from` 隔離, 各應用全量推送互不覆蓋.
- ✅ 空 rows 安全閘防止外部誤送清空權限源 — 清源須有意為之 (推非空集合或走後台), 不會手滑釀災.
- ✅ 四表共用同一全量取代路徑, 差異僅 `ds[keyTable].keys` (schema), 維護單一邏輯.
- ⚠️ 因此**不支援以同步 API 清空某 `from` 的所有列**; 真要清空須走其他路徑. 屬刻意限制.
- ⚠️ 註: `流程_外部應用同步權限資料.md`「欄位邊界」表曾有一行誤記「允許空陣列」, 與實際 `isearr` 行為 (拒絕) 相悖; 以本 ADR 與該文件 E2E-002 / spec 規則摘要為準 (rows 須非空).

**Affected**:
- server/WWebPerm.mjs:652-696 (parsePayload, isearr 非空檢查 :683), :700-744 (syncAndReplaceTabs 全量取代), :1089-1092 (keyTable 白名單)
- server/provideTabs.mjs:8-63, 66-90 (端側欄位白名單)
- spec/流程_外部應用同步權限資料.md (E2E-001~006 / spec 規則摘要)

## ADR-009: 後台寫入採「整表批次 diff」(ltdtDiffByKey), 重給 order

**Status**: Accepted (2026-06-09)

**Context**:
- 後台四份清單 (Users / Grups / Pemis / Targets) 的編輯為「載入整表 → 在前端表格增刪改 → 整批存回」. 存回時須把前端整表與 DB 既有列對齊, 算出哪些列要刪/增/改.
- 若每列各打一支 API, 整表存檔會發出大量請求且難保證原子性與順序; 若整表 delete + re-insert (同 ADR-008 同步路徑) 則會丟失既有列的稽核欄位與既有 `order`.
- 不解決會踩到: 存檔效率差、重複 key、order 錯亂、稽核欄位遺失.

**Decision**:
- 統一走 `updateTabItems(keyTable, userId, rows, keyDetect)` (`WWebPerm.mjs:352`): 先 `ltdtmapping` 對齊 schema 欄位、**重給 `order = k+1`** (`WWebPerm.mjs:360-363`, 與 ADR-008 同步路徑沿用外部 order 不同), 再以 `ckKey` 偵測未給/重複的鍵, 然後以 `ltdtDiffByKey(ltdtOld, ltdtNew, keyDetect)` 算出 `del` / `add` / `diff` 三組, 分別經 `procOrm` 寫入 (`WWebPerm.mjs:419-447`).
- **去重鍵依表而異** (`ckKey`, `WWebPerm.mjs:398-417`): `targets` 以 `id`、`pemis`/`grups` 以 `name`、`users` 以 `email` 去重; **`users` 以 `email` 而非 `name` 去重** (`users` 允許重複 name).
- **diff key (`keyDetect`) 依表而異** (`WWebPerm.mjs:455-476`): `targets`/`users` 以 `id`、`pemis`/`grups` 以 `name` 比對新舊列.
- 全程經 `procOrm` 而非裸 `woItems.insert/save/del`, 以自動補稽核欄位 (`WWebPerm.mjs:427, 435, 443` 註).

**Consequences**:
- ✅ 整表一次存回, 由 diff 算出最小變更集 (只刪/增/改有變動的列), 既有列保留稽核欄位.
- ✅ 重給 order 使後台列順序由前端表格順序決定, 確定且可控.
- ✅ 去重/diff 鍵的差異 (users 用 email) 集中於 `updateTabItems` 一處, 不分散.
- ⚠️ 後台寫入 (重給 order、diff) 與外部同步 (沿用 order、全量取代) 為兩套刻意不同的寫入路徑; 新增寫入入口時須選對路徑, 不可混用.
- ✅ **三層防護之後端 mutex 層已落地** (見 ADR-012): 整表批次寫入以 `pmKeyMutex` 按 `updateTabItems:{tableName}` 粒度序列化 (`WWebPerm.mjs:290, 362`), 同表並行寫入排隊、不同表並行, 避免 read-modify-write race 下的 lost-update.

**Affected**:
- server/WWebPerm.mjs:28 (ltdtDiffByKey import), :352-450 (updateTabItems), :453-478 (updateTargets/Pemis/Grups/Users)
- 並行序列化見 ADR-012 (按鈕雙擊三層防護)

## ADR-010: E2E pixel baseline 採 pixelmatch 反鋸齒感知容差, 非 byte-exact

**Status**: Accepted (2026-06-09)

**Context**:
- 後台四清單與關聯編輯之 e2e 以 pixel baseline 守護整體畫面長相. SVG icon (@mdi) / 字型邊緣之次像素 raster 在跨 browser session 不具決定性.
- 若以 `buf.equals` byte-exact 比對, 肉眼等同的反鋸齒 noise 會被當失敗, 造成無止境 flake (對齊全域 §6.2 殷鑑: icon 改 SVG 後 byte-exact 炸出大量次像素 flake).
- 不解決會踩到: e2e baseline 反覆假性失敗、被迫退化成純語意斷言或頻繁 regen.

**Decision**:
- 比對統一走 `assertBaselineMatch(buf, baselinePath, label, opts)` (`test/e2e-setup.mjs:430`): 以 `pixelmatch` (`includeAA:false` 自動忽略反鋸齒邊緣像素) + `maxDiffPixels` 容差 (預設 100) + `threshold` (預設 0.1) 比對 (`e2e-setup.mjs:473-474`). **舊的 `buf.equals` byte-exact 比對已完全移除, 現行程式碼內不存在**(唯一的 `.equals()` 在 `e2e-setup.mjs:294`, 屬 `captureStable` 判斷連續兩張截圖是否已 settle, 與 baseline 比對無關, 勿誤認).
- 尺寸不同直接 fail (版面/裁切變必為真差異, `e2e-setup.mjs:465-467`).
- 容差只放補強層, 主驗證仍為語意斷言 (DOM 文字 / URL / 元素存在); 真 regression (icon 換 / 版面位移 / 顏色變) 量級遠超 `maxDiffPixels` 仍被抓 (對齊全域 §6.2 原則 1).
- 截圖前以 `captureStable` (`e2e-setup.mjs:227`; 帶紅框標注版為 `captureStableWithBox`, `e2e-setup.mjs:306`) 等 WDrawer nav 收斂、凍結 SVG SMIL 動畫、等 web fonts 載入、retry-until-stable; 遮罩 (`maskRegions`) 僅用於 DOM 凍不到的 `<img>` 內 SVG `<animate>`, 絕不遮靜態 UI.

**Consequences**:
- ✅ 反鋸齒 noise 不再造成 flake, baseline 穩定可重現.
- ✅ 真 regression (量級遠超容差) 仍被抓, 守護價值不減.
- ✅ 比對門檻可由個別 case 經 `opts.maxDiffPixels` / `threshold` 覆寫, 需更嚴/更鬆時有出口.
- ⚠️ 極小幅真差異 (≤ maxDiffPixels 的真改動) 可能漏抓; 屬語意斷言為主、pixel 為補強層的刻意取捨.

**Affected**:
- test/e2e-setup.mjs:14 (pixelmatch import), :163-242 (captureStable), :309-356 (assertBaselineMatch)
- test/e2e-*.test.mjs (所有 e2e 檔)
- test/pics/<flow>/ (baseline 圖檔)

## ADR-012: 按鈕雙擊三層防護 — 元件 promiseUnlock / 頁面 loading / 後端 mutex

**Status**: Accepted (2026-06-18)

**Context**:
- 後台四份清單的「存檔」屬有副作用寫入 (整表 diff 增刪改, ADR-009). 使用者雙擊、或 API 被直打 / e2e race 並發, 都可能讓同一表的 read-modify-write 互相覆蓋 (lost-update): 兩請求各拿同一舊快照算 diff 各自寫, 後者蓋掉前者.
- 單靠前端防護擋不住繞過前端的並發 (API 直打); 單靠後端鎖則使用者仍會看到按鈕無回饋的雙擊體驗.
- 不解決會踩到: 重複寫入 / lost-update、按鈕無鎖視覺、並發下 DB 終態不確定.

**Decision** (對齊本專案 CLAUDE.md「按鈕雙擊防護」與全域 §5.1):
- **元件層**: 四個主存檔鈕 `WButtonCircle` 加 `:promiseUnlock="true"` (`LayoutContent{Users,Grups,Pemis,Targets}.vue:188`); handler 收 `{ pm }`、**第一行 `msg.pm.resolve()`** 釋放視覺鎖、fire-and-forget 呼叫內部 `doSaveXxx`. Ve 系列彈窗 `clickSave(msg)` 亦把 `msg.pm.resolve()` 移到第一行.
- **頁面層**: `doSaveXxx` 走 §5.1 canonical 五段 — 同步檢測 (空列表 / isError) 在 `updateLoading(true)` **之前** early-return; 確定打 API 才開 loading; 各 async 各自 catch + `okSave` 旗標短路; 外層 `core().catch` 接非預期例外、`finally` 一處關 loading. 可預期錯誤走 `$dg.showCheckYes` DOM 模態 (不用原生 dialog).
- **後端層**: `pmKeyMutex` 單例 (`WWebPerm.mjs:290`) 把 `updateTabItems` (`:362`) 與 `syncAndReplaceTabs` (`:714`) 包進 `kmx('updateTabItems:'+keyTable, …)` / `kmx('syncAndReplaceTabs:'+keyTable, …)`. **粒度 = `{op}:{tableName}`** (非 sso 的 per-row key, 因 perm 是整表批次 diff): 同表寫入排隊序列化、不同表並行.

**Consequences**:
- ✅ 三層各擋一類並發: 元件擋 sync 雙擊、頁面 loading overlay 擋 async 期間互動、後端 mutex 擋 API 直打 / e2e race; 同表寫入序列化, 無 lost-update.
- ✅ 後端向後相容, 不改 RPC 介面與回傳; 元件僅加行為 prop, 不動視覺 prop, 不影響 baseline.
- ⚠️ mutex 為 in-process (單後端實例內有效); 多後端實例共用同一 DB 的並發不在此契約內 (對齊全域 §3.1: 多 process 並發屬合約外, 非當前需求).

**Affected**:
- src/components/LayoutContent{Users,Grups,Pemis,Targets}.vue:188 (:promiseUnlock), saveXxx handler + doSaveXxx 五段
- src/components/Ve{Cgrups,Cpemis,Crules,GrupBlngUsers,PemiBlngGrups}.vue (clickSave pm.resolve 第一行)

## ADR-013: 整表批次寫入 (updateTabItems del→insert→save) 採非交易, 不做 rollback

**Status**: Accepted (2026-06-22)

**Context**:
- `updateTabItems` (`server/procCore.mjs`) 以「差異後 del → insert → save」三步落地整表 diff (ADR-009), 三步各自 `await procOrm(...)`, 底層 `w-orm-lmdb` 為各自獨立 commit, **步驟間無交易/rollback**.
- 整段已包在 `kmx('updateTabItems:'+keyTable, …)` 互斥內 (ADR-012): kmx 保證「同表不並發」(防 lost-update), **但不保證「單次操作中途失敗時的部分回滾」**.
- 理論上若中途某步 reject (如 del 成功、insert 因 ORM I/O 失敗), DB 會停在部分更新的不一致態.

**Decision**:
- 維持非交易設計, **不引入跨步交易/補償回滾**. 理由 (對齊全域 §3.1 真痛三條件):
  1. 寫入前 `ckKey` 已擋掉空值/重複等可預期錯誤, diff 後的 del/insert/save 多為純資料操作, 中途失敗需 DB 層 I/O 異常 — **未被觀察** (無重現、無 ticket).
  2. 無任何 spec 宣稱整表 diff 寫入須具交易原子性; lmdb 逐表逐法呼叫本就無跨呼叫交易語意.
  3. 後果 (部分更新) 為推測性, 且本工具為單一管理員驅動之後台, 觸發面窄.
- 失敗時各 `updateXxx` 已 `srLog.error` 記 err key (event=`updateTargets-error` 等) 後 rethrow, 供事後追查.

**Consequences**:
- ✅ 不為消除假想風險引入交易層複雜度與退化 (§3.1).
- ⚠️ 若日後實機觀察到「del 成功 / insert 失敗」導致整表部分遺失 (condition「已被觀察」成立), 再評估最小修法: 將 del+insert+save 包進單一錯誤邊界、失敗時以前置快照補償.

**Affected**:
- server/procCore.mjs (updateTabItems del/insert/save 三步; updateTargets/Pemis/Grups/Users 失敗記 err key)

## ADR-015: 寫入通道授權契約 — 內建 ORM/kpFunExt 由 verifyConn(checkToken) 守門, 整合者負責 getUserByToken 把關

**Status**: Accepted (2026-06-23) — 文件化既有設計契約(對應審計 v1 資安-1 / 待修 T01)

**Context**:
- 本套件對外有兩類通道, 守門不對稱:
  1. **自訂讀取 API**(`/api/getPerm`、`/api/getUserByToken`): 經 `getAndVerifyClientUser` → 呼叫整合者提供之 `verifyClientUser(user)`(範例為 `isAdmin==='y'`), 故套用整合者之 client 授權政策。
  2. **內建 ORM exec 通道 + kpFunExt 變更函式**(`updateTargets/Pemis/Grups/Users`、`select/insert/save/del/delAll`): 唯一守門為 `verifyConn`, 而 `verifyConn` 走 `p.checkToken` → 只驗「token 可經 `getUserByToken` 解析出結構合法 user」(id/email/name/isAdmin 齊備, isAdmin∈{'y','n'} 皆可), **不呼叫 `verifyClientUser`、不檢查 isAdmin**。
- `verifyConn` 為套件**內部**邏輯(非整合者傳入參數), 整合者無法直接改其判定; 整合者對寫入通道的實際控制點是 **`getUserByToken`**(決定哪些 token 能解析出有效 user)。

**Decision**(契約釐清, 不改 code):
- 內建寫入通道之存取邊界 = **「token 可由 `getUserByToken` 解析出結構合法 user」**。**整合者負責**確保 `getUserByToken` 只對「應具寫入權」之主體回傳有效 user(例如僅對管理員 token 解析成功, 或在 `getUserByToken` 內依角色決定)。
- **明確警示(footgun)**: `verifyClientUser` **只套用於上述自訂讀取 API, 不套用於內建寫入通道**。整合者若提供 `verifyClientUser`(如限 isAdmin)並預期它守住一切, 須知**寫入通道不經此 hook** —— 讀取被擋、寫入仍放行。要 admin-only 寫入, 須在 `getUserByToken` 層把關。

**Consequences**:
- ✅ 維持套件彈性: 寫入授權政策交由整合者之 `getUserByToken` 決定, 不在套件硬編特定角色規則。
- ⚠️ 不對稱為已知契約: 若整合者同時(a)以 `getUserByToken` 解析非管理員 token(供讀取 API 經 `verifyClientUser` 過濾)且(b)未在 `getUserByToken` 層限制寫入主體, 則該非管理員 token 可經內建通道寫入全部權限表。此為**整合契約責任**, 非套件缺陷; 但部署前務必確認 `getUserByToken` 的把關範圍。
- 🔧 可選強化(整合者要求時再做, 預設不做, 因有破壞性): 將 `verifyConn` 由 `p.checkToken` 改走 `getAndVerifyClientUser`, 使寫入通道亦套 `verifyClientUser`。**代價**: `getAndVerifyClientUser → getTokenUser` 會額外要求「該 user 須存在於 perm `users` 表且 `isActive='y'`」(`WWebPerm.mjs:441`), 改變整合契約(連線主體須先在 perm 表), 可能破壞既有整合, 故須整合者明確同意後才套用。

**Affected**:
- server/WWebPerm.mjs:1066-1101(verifyConn)→ :1082(p.checkToken)→ :467-477(checkToken); :481-502(getAndVerifyClientUser, 僅用於讀取 API)

## ADR-016: token 經 URL query string 傳遞 — 為整合者轉址需求維持(刻意設計)

**Status**: Accepted (2026-06-23) — 業主確認之刻意設計

**Context**:
- 本套件之登入與對外查詢採 token 經 URL query 傳遞:應用啟動可由網址 `?token=` 取得(App.vue,亦 fallback 至 localStorage);對外 SDK(`perm.mjs` / `getPerm` / `getPermUserInfor`)之端點 URL 含 `?token={token}`(及 getPermUserInfor 之 `&userId=`)。
- **業主整合需要「轉址(redirect / SSO 回跳)」攜帶 token** —— 轉址流程以 URL 帶 token 為必要手段,故維持此設計。
- 一般資安觀點對「token 經 query string」會提示 referrer / 瀏覽紀錄 / log 之揭露面;但須區分:此為**整合契約所需之刻意取捨**,非疏漏。

**Decision**:
- **維持 token 經 URL 傳遞**,不改為純 header/cookie。理由:整合者之轉址/SSO 回跳依賴 URL 攜帶 token,改動會破壞整合流程。

**Consequences**:
- ✅ 滿足整合者轉址/SSO 回跳之必要傳遞方式。
- ✅ 揭露面已部分受控:**`srLog` 全程不記 token 值**(已查證,WWebPerm.mjs 各 srLog 呼叫僅記 event / err key / ip / origin / referer,無 token 欄位),故無「token 明文入持久化 log」之面。
- ⚠️ 已知殘留揭露面(接受):URL 中之 token 可能進入瀏覽器歷史 / referrer header。屬轉址設計之固有取捨;若整合者部署環境敏感,可於其轉址層自行縮短 token 壽命或改用一次性 token,但非本套件預設行為。

**Affected**:
- src/App.vue(?token= 登入)、src/perm.mjs / src/getPerm.mjs / src/getPermUserInfor.mjs(URL 帶 token)、server/WWebPerm.mjs(srLog 不記 token)

## ADR-017: 測試檔命名與定位之三個刻意例外 (e2e-doubleclick / api-updateTabs / getUserRules)

- 盤查 agent 若「發現」此三者命名不一致, 應直接引本 ADR 跳過, 不再列為缺漏。

**Consequences**:
- ✅ 避免為滿足命名對稱而把契約回歸測試硬塞進流程文件, 污染 spec 的使用者視角。
- ⚠️ 代價: 光看檔名無法區分 `e2e-doubleclick`(API-level) 與其餘 9 支 e2e(browser-level)。以檔頭註解與本 ADR 補償。

**Affected**:

## ADR-018: token 失效之模擬走前端狀態 API, 為 act 階段之技術限制例外

**Status**: Accepted (2026-07-09)

**Context**: `e2e-{targets,pemis,grups,users}.test.mjs` 之 **E2E-010(儲存失敗)** 皆需讓「持有無效 token」成為前置狀態, 再點 Save 觀察失敗 modal。四檔一律於 `run()` 中段以 `page.evaluate(() => window.$vo.$ui.updateUserToken('invalid-token-e010'))` 注入(如 `test/e2e-users.test.mjs:359`)。

依全域 e2e 技能, setup helper「只能用於**不是本測試重點**的 trigger」。本案之 token 失效**正是該 case 的重點觸發條件**, 嚴格說屬邊界。惟:

- **無任何真實 UI 路徑可使 token 失效** —— token 由整合者之 `getUserByToken` 判定, 前端無「使 token 過期」之按鈕; 等待真實過期不可行(無 TTL 控制)。
- 該注入**不繞過任何 production 程式碼路徑**: 注入後仍完整走「點 Save → 五段 core() → `$fapi.updateXxx` → 後端 `verifyConn` → `checkToken` 拒絕 → 失敗 modal」, 前後端鏈路一次不少。被抹平的只有「token 如何變成無效」這個前置狀態。

**Decision**: **接受此例外**, 不視為 act 層級違規。理由是它抹平的是**前置狀態(世界已是某狀態)**而非**使用者動作**; 且該注入點之後的整條使用者動作鏈與斷言鏈皆完整。四檔須維持 `//setup：模擬 token 失效` inline 註解。

**Consequences**:
- ✅ E2E-010 得以驗證真實的「後端拒絕 → 前端失敗 modal」鏈路, 而非假造 modal。
- ⚠️ 代價: 「token 從有效變無效」這一步未經 UI 驗證。此步驟本就不存在對應 UI, 故非覆蓋缺口。
- ❌ **不得**擴大解釋: 除 token 失效外, 其餘 act 階段一律走 user-facing input(L1-L3), 禁用 `.fill()` / `vm.method()`。

**Affected**:
- test/e2e-{targets,pemis,grups,users}.test.mjs 之 E2E-010(各檔 `updateUserToken` 呼叫點)

## ADR-019: 語系切換之 e2e —— 移除「假測試」, 改以 server 注入語系覆蓋

**Status**: Accepted (2026-06, 業主拍板; 2026-07-09 補記入檔)

**Context**: 早期存在 `startup-{eng,cht}-E2E-002-lang-switch` 標準圖與對應 case。業主檢視時發現 **`startup-cht-E2E-001-login-ok.png` 與 `startup-cht-E2E-002-lang-switch.png` 兩張標準圖完全相同** —— 該 case 宣稱驗證「切換語系」, 產出的畫面卻與未切換時一致, 即**它什麼都沒驗到**(假測試: 凍結了錯誤行為為真理, 對齊全域 §14.2「測試是規格的翻譯, 不是現狀的指紋」)。
