# 權限規則關聯流程

## 觸發

管理員於權限清單頁（`LayoutContentPemis.vue`）開啟編輯模式（`modeEdit` 開關，`isEditable=true`）後，每一列的 `crules` 欄渲染為一個按鈕（顯示「使用共 N 項對象」或「無使用任何對象」摘要文字）。點該按鈕觸發 `$dg.showVeCrulesById(props.row.id)`（`LayoutContentPemis.vue:225`），由 `showVeCrulesById`（`LayoutContentPemis.vue:1128`）依 row id 找到對應 pemi 列後，開啟權限規則關聯編輯對話框（`VeCrules.vue`，`LayoutContentPemis.vue:1160`）。

本對話框用途為編輯該權限（pemi）對各「管控對象」（target）的授權規則：`crules` 欄位以 JSON5 物件 `{targetId:'y'|'n'}` 形式記錄每個 target 是否被此權限啟用。對話框列出 store 中全部 `targets`（`VeCrules.vue:243,802`），每列一個 target、以 checkbox 表示啟用（`'y'`）或停用（`'n'`），提供全部啟用 / 全部停用 / 反向切換 / 刪除勾選對象等批次操作。

對話框「儲存」僅將編輯結果序列化為 `crules` 字串並 `resolve` 回權限清單頁（`VeCrules.vue:734`），由權限清單頁寫回該列的 `crules` 欄、`refresh`、設 `isModified=true`（`LayoutContentPemis.vue:1168,1172,1175`）；DB 實際寫入須由權限清單頁的 Save（`savePemis` → `updatePemis`）統一持久化。對話框本身不直接打 API。

是否可編輯由開啟時帶入的 `isEditable` 決定（`VeCrules.vue:778,814`）：`isEditable=false` 時為展示模式（標題顯示 `pemiEditCrulesForDisplay`，checkbox `disabled`，無功能區與儲存鈕）。

## 欄位邊界

| 欄位 | 對話框內處理 | 回填 / 持久化 |
|---|---|---|
| `id`（target 主鍵） | 來自 store `targets`，逐 target 一列、唯讀顯示（路徑式 id，如 `專案A/頁A/區塊A`，`targets.mjs:68`）；`isEditable` 時 id 欄理論可編但實際僅作識別 | 作為序列化 key：`kpRule[id]=enable`（`VeCrules.vue:718`） |
| `enable` | checkbox 顯示 `'y'` 為勾選、其餘為未勾（`VeCrules.vue:131`）；可逐列 toggle 或全部啟用 / 全部停用 / 反向切換；`isEditable=false` 時 checkbox `disabled` | 序列化為 `kpRule[id]` 的值（`'y'` / `'n'`，`VeCrules.vue:718`） |
| `crules`（權限列欄位） | 開啟時 JSON5 解析為 `kpRule` 對照各 target 列 enable（`VeCrules.vue:791`）；序列化時以 `JSON.stringify(kpRule)` 產生（`VeCrules.vue:723`） | resolve 回權限頁寫回 `opt.rows[kr].crules`（`LayoutContentPemis.vue:1168`），最終隨 `updatePemis` 持久化 |
| 對話框 `isModified` | 任一 toggle / 全選 / 反選 / 刪除即設真（`VeCrules.vue:548,587,626,669,475`）；控制儲存鈕是否顯示（`VeCrules.vue:11`） | 不持久化，僅控制對話框內儲存鈕顯示 |
| 權限頁 `isModified` | 對話框 resolve 回填後設真（`LayoutContentPemis.vue:1175`） | 控制權限頁 Save 按鈕顯示；按 Save 才打 `updatePemis` 寫 DB |

| 開啟對話框時取不到 pemi 資料（呼叫端） | `pemiEditCrulesNoPemiData` | `$alert` toast | 非預期錯誤 |

> 粒度觀察（對齊全域 §6.2）：本對話框 enable 切換無前端欄位驗證錯誤（target id 取自 store、enable 僅 y/n 二值），錯誤訊息僅在「取不到 target id / 資料」這類非預期情況以 `$alert` 呈現。回填後權限列 crules 欄摘要為**粗粒度**——僅顯示啟用項數（`pemiRnderCrulesHasNRule` 的 `{nms}` 刻意留空，因 target 路徑式 id 過長，`LayoutContentPemis.vue:1079`），測試斷言摘要時應斷言啟用「數量」對應的句，不應期待列出個別 target id。

## spec 規則摘要

- **流程角色**：本對話框為「回填器」——只負責把該權限對各 target 的啟用規則編成 `{targetId:'y'|'n'}` 字串 resolve 回權限清單頁；DB 寫入不在本流程，須由權限清單頁 Save（`savePemis` → `updatePemis`）統一持久化。
- **資料模型**：`crules` 為 JSON5 物件字串，key 為 target 的路徑式 id（`targets.mjs:16` 註「儲存對象路徑，不同階層用句點分開」，樣本 id 為斜線路徑如 `專案A/頁A/區塊A`），value 為 `'y'`（啟用）或 `'n'`（停用）。
- **targets 來源**：對話框列出的 target 一律取自 store `targets`，逐 target 一列；序列化時僅以實際存在的 target id 為 key（不存在的 target 不會憑空產生列）。
- **互動模型**：對話框內 toggle / 全部啟用 / 全部停用 / 反向切換 / 刪除勾選對象只改 `opt.rows` 並設 `isModified=true`，**不即時序列化**；按儲存才於 core() 一次序列化並 resolve。
- **儲存 / 取消分支**：
  - 儲存 → `clickSave` core() 序列化 → `pm.resolve(crules)` → 權限頁回填、`refresh`、權限頁 `isModified=true`。
  - 取消 → `clickClose` → `pm.reject('close window')` → 權限頁 `.catch` 吞掉、不回填、不變更。
- **編輯權限**：`isEditable=false`（展示模式）時無功能區與儲存鈕、checkbox `disabled`，僅供檢視。
- **契約**：`clickSave` 採 `core() + finally`，`finally` 一處關 loading 並通知儲存按鈕解除 loading（`VeCrules.vue:741,744,747`）。
- **粒度**：對話框無欄位驗證錯誤；非預期取不到 target / pemi 之 id 或資料時以 `$alert` toast 呈現；權限列 crules 欄摘要為粗粒度（僅啟用數量，不列個別 target id）。
