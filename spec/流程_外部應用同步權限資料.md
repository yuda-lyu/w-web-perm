# 外部應用同步權限資料流程

## 觸發

外部應用程式（node 端，非瀏覽器使用者）以 app token 對伺服器發起 `POST /syncAndReplaceTabs`，將其本地維護的某一張權限表（`targets` / `pemis` / `grups` / `users` 四選一）的整批列資料推送至伺服器，由伺服器以「依來源 `from` 全量取代」方式覆寫該應用在該表中的資料。

`provideTabs(url, keyTable, from, rows)`（`server/provideTabs.mjs:66`）為外部應用端的呼叫輔助函式：依 `keyTable` 取對應欄位白名單 `ks_*`（`server/provideTabs.mjs:8-63`），以 `ltdtpick` 僅保留白名單欄位（`server/provideTabs.mjs:83`），組成 `{ from, rows }` payload，再以 `axios` POST 至 `url`（範例 `http://localhost:11006/syncAndReplaceTabs?keyTable={keyTable}&token={token}`，`server/provideTabs.mjs:67`）。`keyTable` 與 `token` 由 query 帶入，`from` / `rows` 由 JSON body 帶入。

伺服器端點僅以 query `token` 經 `getAndVerifyAppUser`（`server/WWebPerm.mjs:508`）守門：以 `getUserByToken` 由 token 取出使用者，再以部署方提供的 `verifyAppUser(user, from)` 判定是否為合法 app 使用者；任一步失敗即 reject、回傳 `state='error'`。

## 重要流程

本流程為 server-side app API，**回傳結構為 `{ state, msg }`（`state='success'` / `'error'`），無 UI 終態、無 baseline 圖**。E2E 案例皆為 API 斷言，驗端點回傳值與四表 DB 的全量取代副作用。

- **E2E-001**
  - title: 合法 app token 推送 users 表整批列全量取代成功
  - description: 驗證外部應用以合法 app token 對 `keyTable=users` 推送一批 rows 後，伺服器先依 payload 的 `from` 刪除該應用既有 users 列、再整批 insert 推送資料，回傳 `state='success'`，DB `users` 表中該 `from` 的列被替換為推送內容、且稽核欄位由後端自動補實值。屬全量取代 golden 成功案例。
  - flow:
    - 測試資料：base seed 四表；本案以特定 `from`（如 `appA`）預置數筆 users 舊列，再推送同 `from` 的新批 rows（與舊列不同）。
    - 操作：以合法 app token 對 `/syncAndReplaceTabs?keyTable=users&token=<appToken>` POST `{ from:'appA', rows:[...] }`。
    - 驗證：
      1. 語意：回傳 `state='success'`；DB `users` 表 `from='appA'` 的列等於推送 rows 集合（舊列已不存在）；其他 `from` 的列不受影響；新列 `userId` / `timeCreate` 已由後端補實值。
    - 雙語：本案為 API 回傳值斷言，回傳 `state` / `msg` 非 i18n，單輪即可。
    - 清理：刪除本案 `from='appA'` 推送的列，還原 base seed。

- **E2E-002**
  - title: 推送空 rows 被拒（不支援以空集合清源）
  - description: 驗證以合法 app token 推送 `rows=[]`（空陣列）時，伺服器於 `parsePayload` 的 rows 檢查（`isearr`，要求**非空**陣列，`WWebPerm.mjs:577`）即 reject，回傳 `state='error'`、`msg` 為 `invalidRowsInPayload`，**不進入全量取代（未 delAll）**，故該 `from` 既有列不被清空。此 `isearr` 檢查為「防外部應用誤送空陣列而整批清空資料」之安全閘，故**不支援「以空集合清源」**。屬 payload rows 非空驗證之邊界案例。
  - flow:
    - 測試資料：base seed；以特定 `from` 預置數筆 grups 舊列。
    - 操作：以合法 app token 對 `/syncAndReplaceTabs?keyTable=grups&token=<appToken>` POST `{ from, rows:[] }`。
    - 驗證：
      1. 語意：回傳 `state='error'`、`msg` 為 `invalidRowsInPayload`；DB `grups` 表該 `from` 的列**未被清空**（仍為先前列數）；其他 `from` 的列不受影響。
    - 雙語：API 回傳值斷言，單輪即可。
    - 清理：本案被拒未寫 DB，無需額外清理。

- **E2E-003**
  - title: 無效或缺 token 被守門拒絕
  - description: 驗證未帶 token、或帶不存在 / 無法經 `verifyAppUser` 通過的 token 時，端點於 `getAndVerifyAppUser` 守門階段 reject，回傳 `state='error'`、`msg` 為權限相關字串，DB 四表均不變動。屬 app token 守門拒絕案例。
  - flow:
    - 測試資料：base seed；不預置特化列。
    - 操作：對 `/syncAndReplaceTabs?keyTable=users` 不帶 token（或帶無效 token）POST 合法 `{ from, rows }`。
    - 驗證：
      1. 語意：回傳 `state='error'`，`msg` 為權限相關 key（如未帶 token 時為 `tokenNoPermission`；帶無效 token 經 `getAndVerifyAppUser` 拒絕時為 `cannotFindUserFromToken` 或 `userNoPermission`）；DB `users` 表無任何刪除 / 新增。
    - 雙語：API 回傳值斷言，`msg` 非 i18n，單輪即可。
    - 清理：本案被拒未寫 DB，無需清理。

- **E2E-004**
  - title: 不在白名單的 keyTable 被拒絕
  - description: 驗證 query `keyTable` 不屬 `targets` / `pemis` / `grups` / `users` 四者之一時，端點於 keyTable 白名單檢查階段 reject，回傳 `state='error'`、`msg` 為 keyTable 無效字串，不進入任何表的刪除 / 新增。屬 keyTable 白名單守門案例。
  - flow:
    - 測試資料：base seed；不預置特化列。
    - 操作：以合法 app token 對 `/syncAndReplaceTabs?keyTable=foobar&token=<appToken>` POST 合法 `{ from, rows }`。
    - 驗證：
      1. 語意：回傳 `state='error'`，`msg` 為 `invalidKeyTable`；DB 四表均無變動。
    - 雙語：API 回傳值斷言，`msg` 非 i18n，單輪即可。
    - 清理：本案被拒未寫 DB，無需清理。

- **E2E-005**
  - title: payload 缺 from 或 rows 非陣列被拒絕
  - description: 驗證 payload 缺 `from`（或 `from` 非有效字串）、或 `rows` 非陣列時，端點於 `parsePayload` 解析階段 reject，回傳 `state='error'`、`msg` 為 payload 欄位無效 key，不進入全量取代。屬 payload 結構驗證守門案例。
  - flow:
    - 測試資料：base seed；不預置特化列。
    - 操作：以合法 app token 對 `/syncAndReplaceTabs?keyTable=users&token=<appToken>` POST 缺 `from` 的 body（如 `{ rows:[...] }`）；另一輪 POST `rows` 非陣列的 body（如 `{ from:'appC', rows:'x' }`）。
    - 驗證：
      1. 語意：兩輪皆回傳 `state='error'`，`msg` 分別為 `invalidFromInPayload` / `invalidRowsInPayload`（缺 payload 物件時為 `invalidPayload`）；DB 四表均無變動。
    - 雙語：API 回傳值斷言，`msg` 非 i18n，單輪即可。
    - 清理：本案被拒未寫 DB，無需清理。

- **E2E-006**
  - title: 推送 targets 表全量取代成功（驗四表共用同一路徑）
  - description: 驗證以合法 app token 對 `keyTable=targets` 推送一批 rows，伺服器以與 users 相同的「依 from 全刪後整批 insert」路徑覆寫該應用既有 targets 列，回傳 `state='success'`。本案與 E2E-001 對照，確認四表共用同一全量取代邏輯、僅欄位 schema（`ds[keyTable].keys`）不同。
  - flow:
    - 測試資料：base seed；以特定 `from`（如 `appD`）預置數筆 targets 舊列。
    - 操作：以合法 app token 對 `/syncAndReplaceTabs?keyTable=targets&token=<appToken>` POST `{ from:'appD', rows:[...] }`。
    - 驗證：
      1. 語意：回傳 `state='success'`；DB `targets` 表 `from='appD'` 的列等於推送 rows 集合；推送列僅保留 `targets` schema 欄位（`ltdtmapping` 對齊）。
    - 雙語：API 回傳值斷言，單輪即可。
    - 清理：刪除本案 `from='appD'` 推送的列，還原 base seed。

## 欄位邊界

| 概念 | 處理 |
|---|---|
| `keyTable` | query 帶入；須屬白名單 `[targets, pemis, grups, users]`，否則 reject（`WWebPerm.mjs:998-1002`） |
| `token` | query 帶入；經 `getAndVerifyAppUser` 守門（`getUserByToken` + `verifyAppUser`），非經 verifyConn / Bearer（`WWebPerm.mjs:508-541`） |
| `from` | payload 帶入；須為有效字串。`syncAndReplaceTabs` 以 `from` 為全量取代的範圍鍵：`delAll({ from })` 只刪該 `from` 的列、insert 前逐列複寫 `v.from = from`（`WWebPerm.mjs:625,634`） |
| `rows` | payload 帶入；須為**非空**陣列（`isearr`），空陣列被拒（`parsePayload` reject `invalidRowsInPayload`，`WWebPerm.mjs:577`），不支援以空陣列清空該 `from`。通過後經 `ltdtmapping` 對齊 `ds[keyTable].keys`（`WWebPerm.mjs:630`） |
| 欄位白名單（端側） | `provideTabs` 以 `ks_targets` / `ks_pemis` / `ks_grups` / `ks_users` 白名單 `ltdtpick` 列資料（`server/provideTabs.mjs:8-63,83`）：四表共有 `id` / `order` / `description` / `from`；`pemis` 多 `name` / `crules`；`grups` 多 `name` / `cpemis`；`users` 多 `name` / `email` / `cgrups` / `isAdmin` / `isActive`；稽核欄位（`userId` / `time*`）一律被白名單排除（端側註解 `provideTabs.mjs:14-17` 對齊，由後端 `procOrm` 補） |
| 稽核欄位 | `userId` / `timeCreate` / `userIdUpdate` / `timeUpdate` 不由 payload 帶入，由後端 `procOrm` 於 insert 時自動補（`WWebPerm.mjs:638`） |
| `order` | `syncAndReplaceTabs` 路徑**不重給** order（`WWebPerm.mjs:624` 註「order 大部分由外部給予, 不能複寫」），沿用 payload 帶入值 |

## i18n 訊息粒度規則

本端點為 server-side app API，回傳 `{ state, msg }` 為**機器可讀值**：`state` 為 `'success'` / `'error'`、`msg` 為 camelCase 錯誤 key（如 `tokenNoPermission` / `invalidKeyTable` / `cannotFindUserFromToken` / `userNoPermission` / `invalidPayload` / `invalidFromInPayload` / `invalidRowsInPayload` / `invalidFrom` / `invalidRows`）。上述 key 於 `server/procLang.mjs` 皆有 eng + cht 定義（`procLang.mjs:207,215,167,179,183,187,191,195,199`），由呼叫端（外部應用）以 key 查 lang 決定如何呈現。故本流程訊息粒度即「一個 reject key 對應一條 procLang 雙語訊息」。

## spec 規則摘要

- **權限**：`/syncAndReplaceTabs` 守門完全由端點 handler 內 `getAndVerifyAppUser`（query `token` → `getUserByToken` → `verifyAppUser`）負責；此自訂端點直接 `server.route` 註冊，**不經 verifyConn / Bearer header 通道**（verifyConn 僅守 orm exec 通道）。token 缺 / 無效 / 取不到使用者 / `verifyAppUser` 回 false 一律 reject。
- **取代模型**：本端點為「依來源 `from` 全量取代」——`syncAndReplaceTabs` 先 `delAll({ from })` 刪該應用既有列，再整批 `procOrm insert`，**非逐列 diff（無 ltdtDiffByKey、無 ckKey 去重）**。各應用以各自 `from` 隔離，互不影響彼此資料。
- **空陣列語意**：`rows=[]` **被拒**（`parsePayload` 以 `isearr` 要求非空陣列，`WWebPerm.mjs:577`），回 `state='error'`、`msg` 為 `invalidRowsInPayload`，**未進 delAll**，故不支援「以空集合清源」；此為防外部應用誤送空陣列而整批清空資料之安全閘。
- **欄位**：
  - 四表共用同一全量取代路徑，差別僅 `ds[keyTable].keys`（schema）與端側白名單 `ks_*`。
  - 端側 `provideTabs` 以白名單 `ltdtpick` 過濾欄位，稽核欄位（`userId` / `time*`）一律排除、由後端 `procOrm` 補。
  - `order` 在此路徑沿用外部帶入值，不重給（與後台瀏覽器寫入路徑會重給 order 不同）。
- **回傳**：成功 resolve → `{ state:'success', msg }`；任一檢查 / 守門 reject → `{ state:'error', msg }`（`msg` 為 camelCase 錯誤 key，如 `tokenNoPermission` / `invalidKeyTable` / `invalidFromInPayload` / `invalidRowsInPayload` / `invalidPayload`），於 `procLang.mjs` 皆有 eng + cht 定義，由呼叫端以 key 查 lang 呈現。
- **契約**：handler 採 `pm2resolve(core)()` 統一將 core 的 resolve / reject 轉 `{ state, msg }`；core 內各檢查階段以 `Promise.reject` 短路後續，僅全部通過才進入全量取代。
